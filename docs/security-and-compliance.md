# セキュリティ・コンプライアンス設計メモ

> 2026-05-20 の議論をまとめたもの。

---

## 1. DBテーブル構成と個人情報の保存場所

### 現状（`conversations` テーブル）

現在、顧客の個人情報は `conversations` テーブルに平文で保存されている。

| カラム | 内容 | PII |
|---|---|---|
| `customer_psid` | Facebook Page-Scoped User ID | ○ |
| `customer_name` | 顧客名 | ○ |

### 問題点

- PSIDはFacebook上のユーザー識別子であり、GDPR上PIIに該当する
- 現状は平文保存のため、アプリ層・DBユーザーが侵害された場合に即座に漏洩する

---

## 2. PII分離設計（検討案）

### 基本方針

`conversations` から個人情報を別テーブルに切り出し、アプリ層で暗号化して保存する。

### テーブル設計

```
conversations
  id
  tenant_id
  page_id
  customer_profile_id (UUID, FK) ← PSIDへの直接参照をやめ内部IDに置き換える
  ...

customer_profiles (PII、アプリ層暗号化)
  id (UUID)
  psid_encrypted   (bytea) ← アプリ層で暗号化
  name_encrypted   (bytea) ← アプリ層で暗号化
  tenant_id
```

### メリット

- GDPR の削除要求（Right to Erasure）は `customer_profiles` の行削除だけで完結する
- `conversations` / `messages` にはPIIが残らない
- `deletion_log` の `psid_hash` パターンと一貫した設計になる

### 暗号化の実装方法

このプロジェクトは既に `page_access_token_encrypted`（bytea）でアプリ層暗号化を採用しているため、同じパターンを踏襲する。

| 方法 | 特徴 |
|---|---|
| **アプリ層暗号化** | 既存パターンと統一。シンプルで管理しやすい |
| pgcrypto | DB内で暗号化。キー管理がDB依存になる |
| Supabase Vault | 最も安全だがVault API経由になり実装コストが高い |

## 3. RDS（Amazon RDS）とセキュリティ

### RDSで向上するもの

- **保存時暗号化**: ディスク全体をAES-256で暗号化（物理盗難・ストレージ漏洩への対策）
- **通信の保護**: TLS強制による中間者攻撃防止
- **ネットワーク隔離**: VPC + Security Groupによる接続制限
- **IAM認証統合**: DBユーザー/パスワード管理の削減
- **監査**: 自動バックアップ、CloudTrail/RDSログ連携

### RDSで解決しないもの

RDSの暗号化は「インフラ層の保護」であり、**アプリ層からは平文で読める**。

- アプリケーションが侵害されてSQLを実行された場合、PSIDも氏名も平文で取得できる
- DBユーザーの認証情報が漏洩した場合も同様

### 結論

RDS導入とアプリ層でのPII暗号化は目的が異なるため、**両方実施するのがベスト**。  
RDSを使っても、セクション2の `customer_profiles` 暗号化の必要性は変わらない。

## 4. 事業者保険・法務（BtoB提供時の検討事項）

個人情報を扱うサービスを他社に提供する場合、技術的対策に加えて以下を検討する。

### 保険

- **サイバー保険（情報漏洩保険）**: 漏洩発生時の損害賠償・インシデント対応費用をカバー。個人情報を扱うBtoBサービスでは特に検討に値する

### 法務・制度対応

- **個人情報保護法**: 取り扱い件数が1000件超の場合は届出義務対象になる可能性がある
- **プライバシーポリシー・利用規約**: 責任範囲の明確化が必要
- **顧客との契約**: データ処理に関する責任範囲（委託者/受託者）を契約で明示する

> 保険・法務の判断は専門の弁護士・社労士に相談すること。
