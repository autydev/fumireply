Messenger E2E 実機テスト

Meta の「サンドボックス的なやつ」で本物のメッセージ送受信テストをやる方法。結論を先に言うと、Messenger Platform に WhatsApp のようなサンドボックスは存在しない。代わりに Development Mode + App Roles + 実テストページ を組み合わせて「本物の API を叩いているが審査前のアプリで動かす」状態を作る。

1. 一言まとめ — 「サンドボックス」はない、Dev Mode がそれ

Meta Messenger Platform には明示的な "Sandbox" モードや専用テスト番号は存在しない。WhatsApp Business Platform にはあるが、Messenger には別物。代わりに次のセットで「実 API + 限定公開」のテストができる。

Messenger PlatformWhatsApp Business Platform
専用 Sandboxなしあり（テスト番号 + 5 件まで宛先登録）
テスト用の代替Development Mode + App Roles で「app に role を持つ人だけが実 API を使える」状態を作るSandbox がそのまま使える
必要な実体本物の Facebook Page（テスト用に作ったページでも可）+ 本物の FB アカウントテスト番号は Meta が貸与、宛先電話番号は手動登録
ネット上の記事で「Messenger サンドボックス」と書いてあるものは、ほぼ全部 Development Mode のことを指しているか、WhatsApp Sandbox との混同。Meta の公式ドキュメントに sandbox という単語は Messenger 文脈ではほぼ出てこない。

2. Meta が提供している実機テスト用の仕組み 4 つ

「サンドボックス」という名前のものはないが、組み合わせるとサンドボックス相当になる仕組みは 4 つある。

2.1 Development Mode（最重要）

App Dashboard 左上の App Mode を Development にしている間は、App に Role を持つアカウントとのやり取りだけが API / Webhook 経由で動く。第三者がそのページに DM を投げても webhook には来ない。これが事実上のサンドボックス境界。

  • 新規作成した App はデフォルトで Development Mode。
  • App Review なしで pages_messaging 等の権限を roled user に対してだけ使える。
  • Live Mode に切り替えるのは App Review 通過後。fumireply の場合 MVP 申請を出すまでは Dev Mode のまま。

2.2 App Roles（Admins / Developers / Testers）

App Dashboard → Roles → App Roles で実 FB アカウントに役割を付与する。Tester は招待 → 相手が承諾、で発効。

  • Admin：自分。App 設定全権。
  • Developer：実装者。Webhook / Token 設定可。
  • Tester：実機テスター用。設定変更不可だが、Dev Mode の app と「実 API で」やりとりできる。

2.3 Test Users（自動生成 FB アカウント）

App Dashboard → Roles → Test Users から「FB 上に存在するが本物のユーザーではない」アカウントを生成できる。fumireply では基本的に使わないのが結論。理由は次のとおり：

  • Test User は Test User 同士か、その Test User が作った Test Page としか messaging の往復ができない（実 Page との DM 往復は安定しない）。
  • fumireply は「実 Page に DM が来る → 返信」を再現したい。本物のページ + 本物のテスターのほうが審査時の挙動と一致する。
  • Test User の用途は「Login Dialog の OAuth フロー単体テスト」「同意ダイアログ表示確認」などに限定。

2.4 Webhook の "Test" ボタン と Graph API Explorer

これは「往復テスト」ではなく「片方向テスト」。

  • Webhooks → Test：App Dashboard の Webhooks 設定画面から、Meta が用意したサンプルペイロードを受信エンドポイントに POST してくれる。X-Hub-Signature-256 も付いた状態でくる。受信 Lambda の署名検証 + DB INSERT までを単独で確認したいときに最強。実 Page も実 DM も不要。
  • Graph API Explorer：POST /me/messages を Page Access Token で叩いて Send API 単独テスト。返信側だけ確認したいときに使う。
Webhooks の Test ボタンが投げるペイロードはダミー PSID なので、そのまま connected_pages 経由で本物の page_id 解決をやろうとすると外れる。本物の page を connect 済みなら、Test ボタンは sender / page を実値に書き換えてから投げる UI になっている（ものもある）。挙動は時期で変わるので「来たペイロードを log に出して中身を確認 → 必要なら手で修正したものを curl で送る」が確実。

3. 登場人物（FB アカウントを 2 つ用意する）

これが地味に詰まるポイント。Page 管理者は自分の Page に DM を投げられない。なので最低 2 つの実 FB アカウントが必要。

A. Page Owner（あなた）

  • App の Admin Role 持ち
  • Test 用 Facebook Page の 管理者
  • 長期 Page Access Token を発行する側
  • fumireply 管理画面に operator@… でログインする想定の人

B. Customer Tester（顧客役）

  • App の Tester Role を承諾済み（Dev Mode 必須条件）
  • Page を一切管理していない別 FB アカウント
  • Messenger アプリ / m.me から Page に DM を投げる側
  • 実機テスト中ずっとブラウザを開いて返信を待つ役
サブ FB アカウントが手元にないなら、家族・同僚に Tester ロールを依頼して借りるか、別ブラウザプロファイル + 別電話番号で用意する。Meta は同一人物の複数アカウントを規約上禁止しているので、本気でやるなら同僚案件。

4. 事前準備 — App / Page / Token / Webhook

fumireply で初回 E2E をやる前に最低限揃えておくものを seed.html の Review seed と並行で準備する。順番が重要。

1
Test 用 Facebook Page を作成。Page Owner（A）の個人アカウントから新規 Page を作る。カテゴリは ビジネス系が無難。Page Username（m.me/<username> で使う）も設定。
↓
2
App Dashboard で Messenger プロダクトを追加。Add Product → Messenger。すでに追加済みならスキップ。
↓
3
Page を App に接続。Messenger 設定 → Access Tokens セクションで Test Page を選んで Add or Remove Pages。Page が一覧に出る。
↓
4
長期 Page Access Token を取得。詳細は meta-tokens.html。Graph API Explorer で短期 User Token → fb_exchange_token で長期 User → /me/accounts で長期 Page Token を抜き出す。
↓
5
Review seed を流して DB 投入。npm run db:seed:review で connected_pages に Page Token を AES-256-GCM で暗号化保存。詳細は seed.html。
↓
6
Webhook URL を Messenger に設定。Messenger → Webhooks → Add Callback URL。URL は API Gateway の /api/webhook、Verify Token は SSM の値。サブスクリプションフィールドは messages, messaging_postbacks の最低 2 つ。
↓
7
Test Page を Webhook 購読対象に追加。同じ画面の Page セレクタで Test Page を選び Subscribe。これを忘れると DM が来ても webhook が飛ばない。
↓
8
Customer Tester（B）を Tester Role に招待。App Dashboard → Roles → App Roles → Add Testers。B のアカウントで通知を承諾。
App Mode が Development であることを最後に必ず確認する。Live Mode に切り替えてしまうと、App Review 未通過の権限で動かしているのが Meta に検知され、警告 → 機能停止のリスクがある。Development Mode のまま B が DM を投げる構成が「実 API + 安全範囲」の正解。

5. E2E 実機テストの実行手順

準備が終わったら以下の流れを実機で 1 周する。これが App Review でレビュワーが再現するフローと同型。

1
あなた（A）は管理画面にログイン。https://<cloudfront-domain>/login → Supabase Auth の operator@malbek.co.jp。受信一覧 (/inbox) を開いて待機。
↓
2
Customer Tester（B）が Page に DM を送る。スマホの Messenger アプリ or m.me/<page-username> から「テストメッセージです」など。
↓ Meta → 30 秒以内
3
Webhook 受信 Lambda が POST を受ける。署名検証 → page_id から tenant_id 解決 → messages テーブル INSERT → SQS enqueue → 200 を返す。CloudWatch で Webhook OK ログを確認。
↓ SQS → Worker
4
AI 下書き Worker が Anthropic を呼ぶ。Claude Haiku 4.5 が返信下書きを生成し、ai_drafts に status='ready' で保存。p95 60 秒以内。
↓ A の画面を更新
5
受信一覧に新着が表示される。スレッドを開くと B のメッセージと AI 下書きがプレースホルダで入った返信欄が見える。
↓
6
下書きを編集 or そのまま送信。送信ボタン → createServerFn 経由で Send API POST /me/messages。応答 200 で messages.send_status='sent' 更新。
↓ 5 秒以内
7
B の Messenger アプリに返信が届く。これで往復完了。受信から返信 UI に反映、送信から相手着信、それぞれ p95 SLA 内に収まっているかを目視確認。

6. 各レイヤで何を観測するか

1 周しただけで OK とせず、各レイヤの「期待されるサイン」を必ず取りに行く。どこか 1 つでも欠けていれば審査でも落ちる。

レイヤ確認するもの確認の仕方
Webhook Lambda 署名検証 OK / 署名検証 NG が両方ログに出ること、X-Hub-Signature-256 の不一致時 403 になること CloudWatch Logs /aws/lambda/fumireply-webhook。一度わざと App Secret を間違えて NG ケースも見ておく
DB（Supabase） messages, conversations, ai_drafts に行が増えていること、tenant_id が正しく入っていること Supabase Studio の Table Editor で SELECT * ORDER BY created_at DESC LIMIT 5
SQS Webhook → Worker の enqueue / dequeue カウントが一致 SQS Console のメトリクス。DLQ に落ちていないこと
Worker Lambda Anthropic API 呼び出し成功、ai_drafts.status が generating → ready CloudWatch Logs /aws/lambda/fumireply-ai-worker
UI 下書きが返信欄プレースホルダに入る、編集して送信ボタンで成功フィードバック ブラウザで実操作。Network タブで createServerFn 呼び出しが 200
Send API Meta の応答に recipient_id と message_id messages.meta_message_id に m_ プレフィクスの ID が入る
受信側 Messenger B の Messenger アプリで返信が届く、5 秒以内 目視。スマホのプッシュ通知を切らない

7. WhatsApp サンドボックスとの違い（混乱しがちなので）

調べているときに出てくる「Meta サンドボックス」記事は WhatsApp 由来のものが多いので、混同しないように整理しておく。

WhatsApp Business Sandbox

  • Meta が貸与する 専用テスト電話番号がある
  • 事前に登録した最大 5 件の宛先電話番号にだけメッセージを送れる
  • App Review なしで messages API が叩ける
  • Production 番号への移行は別フロー

Messenger Platform

  • 専用テストインフラはない
  • 本物の Page を 1 つ作って Dev Mode の App に紐付ける
  • App Roles を持つアカウントとだけ往復できる
  • Live Mode 切り替え = App Review 通過後
fumireply は Messenger 側なので「専用番号もらえる」想定で動くと詰まる。本物の Page を 1 つ用意するのがスタートライン。

9. App Review 提出時のテストアカウント引き渡し

E2E が回るようになったら、その同じ仕組みをレビュワーに使ってもらえる形に整えるのが App Review 提出。fumireply の spec.md User Story 1〜2 がこれ。

  • レビュワー用 FB アカウントは Meta が用意してくれない。レビュワーは自分の FB アカウントで B 役を演じる。fumireply 側は事前にレビュワーアカウントに Tester Role 招待 → 承諾してもらう、または App を Live Mode にする。MVP ではいったん Live Mode + 必要権限を Limited Use で申請するのが普通。
  • 管理画面ログイン用の専用アカウントは fumireply 側で発行して提出フォームに記載（reviewer@malbek.co.jp）。Supabase Auth 側で seed しておく（seed.html 参照）。
  • スクリーンキャスト動画もこの実機 E2E と同じフローを撮る。User Story 4 参照。
つまり「サンドボックス的な何か」の実体は：本物の Page + 本物の FB アカウント B + Dev Mode + Tester Role の組み合わせ。これで「実 API を叩いているが、限られた人にしか影響しない」状態になり、App Review 前でも安全に round-trip テストができる。レビュワーには Live Mode に切り替えた後の同じ仕組みをそのまま見せる。
