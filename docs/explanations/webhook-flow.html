1. 一言まとめ

Meta UI で「コールバックURLまたは認証トークンが確認できませんでした」のエラーが出る状態。原因は Meta が GET ハンドシェイクを叩いたときに 200 が返らないこと。API Gateway 直叩きは 200 で正常、CloudFront 経由のときだけ 403 なので、CloudFront 側だけが問題で残っている。Invalidation で治る可能性が高い。

3. Meta が叩く Webhook には 2 種類ある

同じ URL を 登録時と運用時で違う方法で叩く。Lambda の処理も両方を 1 ハンドラで分岐する形になっている (webhook/src/handler.ts)。

場面HTTP来るもの返すもの
URL 登録 / 更新時の 1 回だけ GET QueryString に hub.mode=subscribe + hub.verify_token + hub.challenge 200 + body hub.challenge の値そのまま (text/plain)
以降ずっと (DM 等のイベント受信) POST JSON body + X-Hub-Signature-256 header 200 (空 body 可)。Meta は 20 秒以内必須
Meta UI の登録画面に「メソッドを選ぶ欄」はない。同じ Callback URL に対して Meta 側が GET と POST を投げ分ける。Lambda 側で両方のルートを API Gateway に登録する必要があるのはそのため。

4. GET 検証ハンドシェイクのシーケンス

Meta UI の「Verify and Save」ボタンを押した瞬間、Meta が裏でやる動き。今これが 200 を返せていないので Meta UI が「URL またはトークンが確認できない」エラーを出している。

Meta CloudFront API Gateway Lambda SSM 処理 / レスポンス
●→ GET /api/webhook?hub.mode=subscribe&hub.verify_token=XXX&hub.challenge=YYY
●→ CloudFront /api/* behavior → APIGW オリジンに転送。query string そのまま forward。
●→ route GET /api/webhook 一致 → Lambda invoke (proxy v2)。
●→ handler 内 handleGet(event)。hub.mode==='subscribe' 確認 → SSM から verify_token 取得。
●← SSM が SecureString を復号して返す。
● 一致: 200 body YYY を返す。
不一致: 403 Forbidden を返す。
←● API Gateway 経由で CloudFront に戻る。
←● CloudFront はキャッシュせず（TTL=0）そのまま返す。
●← Meta が YYY を受信 → 検証成功 → URL 登録完了。
このフローを手元で再現するコマンド: 
TOKEN=$(aws ssm get-parameter \
  --name /fumireply/review/meta/webhook-verify-token \
  --with-decryption --query Parameter.Value --output text)

curl -i "https://review.fumireply.ecsuite.work/api/webhook?hub.mode=subscribe&hub.verify_token=$TOKEN&hub.challenge=test123"

# 期待: HTTP/2 200 + body test123

5. POST イベント受信のシーケンス

登録が通ったあと、実際に DM が来たときの流れ。Meta は 20 秒以内に 200 を返さないと「届かなかった」と判断して再送するため、AI 呼び出しなど重い処理は SQS に逃がす設計。

Meta CloudFront API Gateway Lambda DB / SQS 処理 / レスポンス
●→ POST /api/webhook + JSON body + X-Hub-Signature-256
●→ 同 behavior 経由で APIGW へ転送。
●→ route POST /api/webhook 一致 → Lambda invoke。
● handler 内 handlePost(event)。SSM から App Secret を取得 → HMAC-SHA256 で署名検証。
● 署名不一致: 403 即返却。攻撃者・誤設定の防御線。
●→ 署名一致: page_id → tenant_id 解決 → messages テーブル INSERT → SQS enqueue (AI Worker 用)。
←←● 200 empty body を即返却。p95 < 2 秒、p99 < 5 秒 (Meta の 20 秒 SLA 内)。
●← Meta は 200 を受領 → イベント配信成功 → 同イベントは再送しない。
·· 非同期: AI Worker Lambda が SQS dequeue → Anthropic API → ai_drafts 保存 (status='ready')。
Webhook 受信 Lambda は AI 呼び出しを含めない。Anthropic API は p95 数秒〜数十秒かかるので、含めると Meta の 20 秒タイムアウトに当たって再送が暴発する。SQS で分離しているのはそのため。

6. 現在の症状 — どこで詰まっているか

同じパスを 2 つの経路で叩いた結果が真逆。

200 API Gateway 直叩き

curl -i "$APIGW_URL/api/webhook?hub.mode=subscribe&hub.verify_token=$TOKEN&hub.challenge=test123"

HTTP/2 200
content-type: text/plain; charset=utf-8
content-length: 7

test123

→ Lambda は正しく動作している。SSM 取得・token 比較・200 返却すべて OK。

403 CloudFront 経由

curl -i "https://review.fumireply.ecsuite.work/api/webhook?hub.mode=subscribe&hub.verify_token=$TOKEN&hub.challenge=test123"

HTTP/2 403
x-amzn-errortype: ForbiddenException
x-cache: Error from cloudfront

{"message":"Forbidden"}

→ CloudFront → API Gateway の経路に何か起きている。Lambda には到達していない（CloudWatch Logs に記録なし）。

注目ポイント: {"message":"Forbidden"} + x-amzn-errortype: ForbiddenException は API Gateway HTTP API が「ルート未マッチ」で返すデフォルト応答の形。x-cache: Error from cloudfront から、CloudFront が「オリジンからエラーが返ったので返す」状態であることがわかる（=オリジンが APIGW のエラーレスポンスを CloudFront に返している）。

7. 仮説 A / B

仮説 A: CloudFront に古い 403 が残留している

過去に Lambda が Cannot find module 'handler' で初期化に失敗していた時期に、CloudFront 経由で 403 が返り続けていた。CloudFront はデフォルトで 4xx エラーレスポンスを最大 10 秒間ネガティブキャッシュするが、より長く残るケースもある。Cache-Control / TTL の挙動によっては古いレスポンスが見えてしまう。

確認・治療: create-invalidation でキャッシュを強制クリア（即効性あり）。

仮説 B: CloudFront のオリジン設定が古い API Gateway を指している

terraform apply のタイミングのズレで、CloudFront に登録された APIGW オリジンの DomainName が、現在動いている API Gateway とは別のもの（過去に作られて消された API）を指している可能性。その場合 CloudFront は誰もいないオリジンに転送 → APIGW がデフォルトの「ルート未マッチ」を返す。

確認: aws cloudfront get-distribution で取得した origin と terraform output api_gateway_invoke_url を文字列比較。

8. 次にやることの順序

仮説 A は即効・低リスクなので先に試す。

Step 1: CloudFront キャッシュ強制クリア 
aws cloudfront create-invalidation \
  --distribution-id E1BVZ9FF47RUPT \
  --paths "/api/*"

30 秒〜2 分待ってから再試行:

TOKEN=$(aws ssm get-parameter \
  --name /fumireply/review/meta/webhook-verify-token \
  --with-decryption --query Parameter.Value --output text)

curl -i "https://review.fumireply.ecsuite.work/api/webhook?hub.mode=subscribe&hub.verify_token=$TOKEN&hub.challenge=test123"

200 + test123 なら原因 A 確定 → そのまま Meta UI で Verify and Save。

Step 2: A で治らなかった場合 — オリジン設定の確認 
# CloudFront が見ているオリジン
CF_ORIGIN=$(aws cloudfront get-distribution \
  --id E1BVZ9FF47RUPT \
  --query "Distribution.DistributionConfig.Origins.Items[?Id=='APIGW-fumireply-review'].DomainName | [0]" \
  --output text)
echo "CloudFront origin: $CF_ORIGIN"

# 直叩きで通った API Gateway
cd /Users/ssdef/program/fumireply/terraform/envs/review
APIGW_HOST=$(terraform output -raw api_gateway_invoke_url | sed -E 's|https://||')
echo "Terraform: $APIGW_HOST"

2 値が違う場合:

cd /Users/ssdef/program/fumireply/terraform/envs/review
terraform apply -target=module.static_site

これで CloudFront のオリジンが正しい APIGW に揃う。

Step 3: それでも治らなかった場合 — Lambda ログ確認 
aws logs tail /aws/lambda/fumireply-review-webhook --since 10m

ログに START RequestId:... があれば Lambda には届いている → Lambda 内部の挙動を疑う。ログが空なら経路（CloudFront / APIGW）の問題が継続中。

9. Webhook 登録後にやること

Meta UI で Verify and Save が通ったあとの流れ。詳細は e2e-meta-testing.html §4 以降。

  1. Page を Webhook 購読対象に追加 — Messenger → Webhooks 設定画面で Test Page を選んで Subscribe。Subscription Fields は messages, messaging_postbacks
  2. Customer Tester（B 役）を Tester Role に招待 — App Dashboard → Roles → Add Testers。B のアカウントで通知を承諾
  3. 実機 E2E — B から Page に DM → 受信一覧に表示 → スレッドを開いて AI 下書き確認 → 編集して送信 → B の Messenger に着信
  4. 各レイヤの観測ポイント確認 — CloudWatch Logs / DB の messages, conversations, ai_drafts / SQS メトリクス / Send API レスポンスの meta_message_id
Webhook 登録が通ったあと、POST 用に署名検証が機能するかも追加で curl 検証しておくと安心。webhook.html §署名検証 を参照。署名検証が通っていない POST には Lambda が 403 を返す設計なので、初動でうっかり通る方が事故。
