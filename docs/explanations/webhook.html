Webhook 受信と署名検証

Meta が顧客メッセージをこちらに POST してくる Webhook の認可・検証ロジック。GET (登録時) と POST (イベント受信時) で使うシークレットが違う点が要。

1. 一言まとめ

Webhook エンドポイントは AWS API Gateway + Lambda で /webhook として公開されている。 Meta は 2 つの方法で叩いてくる:

  • GET 登録 (subscribe) のとき 1 回 — Verify Token で「このサーバーは持ち主だ」を証明する
  • POST イベントの度 — App Secret で HMAC した署名 (X-Hub-Signature-256) で「このボディは Meta が作った」を証明する

2. GET と POST、それぞれが使うシークレット

メソッド 使うシークレット SSM パス 意味
GET Webhook Verify Token /fumireply/<env>/meta/webhook-verify-token サブスクリプション登録時の所有確認。Meta が ?hub.verify_token=… を投げてくるので、保存値と一致したら hub.challenge をそのまま echo
POST App Secret App /fumireply/<env>/meta/app-secret HMAC-SHA256 のキーとして使い、ボディから期待署名を再計算して X-Hub-Signature-256 ヘッダと一致するか検証
SSM_PATH_PREFIX について: 上のパスの <env> 部分は実際には SSM_PATH_PREFIX 環境変数で決まる (デフォルト /fumireply/)。 review 環境なら /fumireply/review/、e2e なら /fumireply/e2e/ など。

3. GET — Webhook 登録時のオーナー証明

Meta App ダッシュボードで Webhook URL を設定 (Subscribe) すると、Meta は次の URL を 1 回だけ叩く:

GET /webhook?hub.mode=subscribe
            &hub.challenge=<ランダム文字列>
            &hub.verify_token=<設定した文字列>

サーバーは hub.verify_token が事前共有値と一致するか確認し、一致したら hub.challenge をそのままレスポンスボディに echo すれば登録完了。

webhook/src/handler.ts

async function handleGet(event) {
  const qs = event.queryStringParameters ?? {}
  const mode = qs['hub.mode']
  const challenge = qs['hub.challenge'] ?? ''
  const token = qs['hub.verify_token'] ?? ''

  if (mode !== 'subscribe') return { statusCode: 403, body: 'Forbidden' }

  const ssmPrefix = env.SSM_PATH_PREFIX.replace(/\/$/, '')
  const verifyToken = await getSsmParameter(`${ssmPrefix}/meta/webhook-verify-token`)

  if (token !== verifyToken) return { statusCode: 403, body: 'Forbidden' }

  return { statusCode: 200, body: challenge }            // challenge をそのまま返す
}
Verify Token はテナント横断の共通値: Meta App は 1 個 (= 全テナントの Page が同じ App を経由) を前提とする MVP 設計。 もしテナントごとに別 Meta App を持つ運用にしたければ、テナント単位の SSM パス分割を検討する必要がある。

4. POST — イベント受信時の署名検証

Meta は新着メッセージのたびに次のような POST を投げてくる:

POST /webhook
Content-Type: application/json
X-Hub-Signature-256: sha256=<HMAC-SHA256(appSecret, rawBody) の hex>

{ "object": "page",
  "entry": [{
    "id": "<PAGE_ID>", "time": 1714800000000,
    "messaging": [{
      "sender":    { "id": "<PSID>" },
      "recipient": { "id": "<PAGE_ID>" },
      "timestamp": 1714800000000,
      "message":   { "mid": "...", "text": "こんにちは" }
    }]
  }] }

受信処理の最初の関門が署名検証。失敗したら 401 を返してそのまま終了する。

Meta
POST /webhook (raw body, X-Hub-Signature-256: sha256=…)
Webhook Lambda
SSM: getSsmParameter('…/meta/app-secret')
crypto.HMAC-SHA256(appSecret, rawBody) → expected
timingSafeEqual(expected, provided) ?
false → 401 Invalid signature (ここで終了)
true → JSON parse → ペイロード処理 (下記)

5. HMAC-SHA256 と timing-safe 比較

webhook/src/signature.ts

import { createHmac, timingSafeEqual } from 'node:crypto'

export function verifySignature(rawBody: string, signature: string, appSecret: string): boolean {
  if (!signature.startsWith('sha256=')) return false

  const expected = Buffer.from(
    createHmac('sha256', appSecret).update(rawBody, 'utf-8').digest('hex'),
    'hex',
  )
  const provided = Buffer.from(signature.slice('sha256='.length), 'hex')

  if (expected.length !== provided.length) return false
  return timingSafeEqual(expected, provided)             // 一致しても不一致でも比較時間が同じ
}

なぜ === ではなく timingSafeEqual なのか

通常の文字列比較は最初の不一致バイトで早期 return するため、外部から「最初の N バイトまでは合っている」を時間差で推測できる (タイミング攻撃)。 timingSafeEqual は常に全バイトを XOR で比較するため、一致 / 不一致いずれでも実行時間がほぼ同じ。 HMAC キー (App Secret) が固定ゆえ、攻撃者がオラクルとして使う余地を消す目的。

raw body の取り扱い

HMAC はサーバー側で再計算したバイト列が、Meta が署名した時のバイト列と完全一致することが前提。 そのため、JSON.parse したオブジェクトを再 stringify して計算するのは絶対 NG (空白やキー順が変わる)。 必ず受信した raw body を使う。handler.ts では event.isBase64Encoded も考慮して下のように取り出している: 

const rawBody = event.isBase64Encoded
  ? Buffer.from(event.body ?? '', 'base64').toString('utf-8')
  : (event.body ?? '')
signature ヘッダ名のケース: API Gateway 経由ではヘッダ名が小文字で正規化されるが、念のため x-hub-signature-256 と X-Hub-Signature-256 の両方を読んでいる (handler.ts: 171-172)。

6. ペイロード処理 (テナント解決 → withTenant → SQS)

署名検証を通過した後の流れ。

1
JSON.parse(rawBody) を Zod スキーマでバリデーション。失敗しても 200 を返す (Meta のリトライを抑える) が、内部的にはログに残してスキップ
↓
2
各 entry について entry.id = Page ID を取り出し、dbAdmin (service role) で connected_pages を引いて (tenantId, pageUuid) を解決
※ ここはテナント解決前なので RLS を貫通する必要がある
↓ tenantId が確定
3
withTenant(tenantId, async (tx) => {…}) 内で:
・conversations を upsert (psid 単位)
・messages を onConflictDoNothing() で挿入 (mid ユニーク制約で重複イベント無効化)
・conversations.unreadCount を +1 / lastInboundAt 更新
・テキスト系メッセージなら ai_drafts (status='pending') を入れる
↓ 新しい inbound が確定したときのみ
4
SQS に { messageId } を enqueue。AI Worker Lambda が拾って下書き生成へ
↓
5
200 EVENT_RECEIVED を返す。Meta の 20 秒 SLA に余裕で間に合う設計
echo メッセージ (送信した自分のメッセージが Webhook で返ってくる) の扱い: message.is_echo === true のときは、messages.meta_message_id 一致で send_status='sent' を打つだけで、新規 inbound としては扱わない。これでアプリ側の送信ステータスが確定する。
