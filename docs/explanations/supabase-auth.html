1. 一言まとめ

fumireply は Supabase Auth が発行する JWT を HttpOnly Cookie に格納し、サーバー側ミドルウェアで毎リクエスト検証する方式。 ユーザー DB はアプリ側に持たず、所属テナント (tenant_id) と権限 (role) は Supabase ユーザーの app_metadata に書いておき、JWT クレームから取り出す。 ミドルウェアは取り出した tenantId をリクエストコンテキストに入れ、各 DB 操作は withTenant() 経由で SET LOCAL app.tenant_id し、Postgres の RLS が他テナントの行を見せない、という二段構え。

設計判断 (research.md R-002): ユーザー / セッションテーブルを自前で持たず、Supabase Auth に丸投げ。MVP のスコープでは Cognito より簡素で初期構築が速い、というトレードオフ採用。

2. 取得しているもの一覧

ログイン成功時に Supabase から戻ってくるもの・アプリが保管するものは下表のとおり。

名前 どこに置く 寿命 用途
access_token JWT Cookie sb-access-token HttpOnly 1 時間 (3600s) 毎リクエストで supabase.auth.getUser(token) に渡してユーザー特定。署名検証は Supabase 側。
refresh_token Cookie sb-refresh-token HttpOnly 30 日 (2592000s) アクセストークンが切れたとき refreshSession() で新しいペアと交換。Supabase は使用ごとに自動ローテーション。
user.id (UUID) JWT クレーム / Supabase Auth テーブル — 誰が送信した返信かを messages.sent_by_auth_uid に記録するために使う。
user.app_metadata.tenant_id JWT クレーム (Supabase 側で発行) — 所属テナント UUID。アプリの全 DB アクセスでこの値を使ってテナント分離する (最重要)。
user.app_metadata.role JWT クレーム — 'operator' | 'reviewer' | null。UI / 認可分岐に使う。
「ユーザーテーブル」が無いことに注意: Drizzle スキーマ (app/src/server/db/schema.ts) に users テーブルは存在しない。 ユーザー名簿は Supabase Auth が単独の真実源。アプリ側は JWT を「読む」だけ。

3. ログイン (signInWithPassword)

3.1 シーケンス

1
ブラウザの LoginForm がメール / パスワードを送信し、サーバー関数 loginFn が呼ばれる
↓
2
performLogin() → supabase.auth.signInWithPassword({ email, password })
↓ Supabase が JWT (access) と refresh_token を発行
3
user.app_metadata.tenant_id が無いユーザーは弾く (invalid_credentials として返す)
↓
4
setCookie('sb-access-token', ..., { httpOnly:true, secure:true, sameSite:'lax', maxAge:3600 })
setCookie('sb-refresh-token', ..., { httpOnly:true, secure:true, sameSite:'lax', maxAge:2592000 })
↓
5
{ id, email, tenantId, role } をクライアントに返却 → ブラウザは /inbox へリダイレクト

3.2 重要なコード

app/src/routes/(auth)/login/-lib/login.server.ts

const { data: d, error } = await getSupabaseClient().auth.signInWithPassword({
  email: data.email,
  password: data.password,
})

const tenantId = user.app_metadata?.tenant_id as string | undefined
if (!tenantId) {
  return { ok: false, error: 'invalid_credentials' }   // tenant 未割当ユーザーはログイン不可
}

setCookie('sb-access-token',  session.access_token,  { httpOnly:true, secure:true, sameSite:'lax', path:'/', maxAge:3600 })
setCookie('sb-refresh-token', session.refresh_token, { httpOnly:true, secure:true, sameSite:'lax', path:'/', maxAge:2592000 })
HttpOnly + secure + SameSite=lax: JS から読めない (XSS でトークン盗難不可) / HTTPS 限定 / クロスサイト POST に乗らない (CSRF 緩和)。 Bearer ヘッダ方式ではなく Cookie 方式を選んでいる主動機がここ。

4. 各リクエストの検証 (auth-middleware)

TanStack Start の createMiddleware({ type: 'function' }) でラップしたサーバー関数は、 本体ロジックの前に必ずこのミドルウェアを通る。やっていることは 4 段階。

1
getCookie('sb-access-token') を読む。無ければ /login にリダイレクト
↓
2
verifyAccessToken(token) = supabase.auth.getUser(token) で検証。
失敗したら sb-refresh-token でリフレッシュ → 新トークンで Cookie を上書き
↓ user オブジェクト確定
3
user.app_metadata.tenant_id を抽出。空なら /login?error=no_tenant
tenants テーブルを dbAdmin (service role) で引いて status === 'active' を確認。違えば /login?error=tenant_suspended
↓
4
next({ context: { user: { id, email, tenantId, role } } }) 後段ハンドラに渡す

app/src/server/middleware/auth-middleware.ts

export const authMiddleware = createMiddleware({ type: 'function' }).server(
  async ({ next }) => {
    const accessToken = getCookie('sb-access-token')
    if (!accessToken) throw redirect({ to: '/login' })

    let user = await verifyAccessToken(accessToken)

    if (!user) {                                            // 期限切れ → refresh
      const refreshToken = getCookie('sb-refresh-token')
      if (!refreshToken) throw redirect({ to: '/login' })
      const refreshed = await refreshSession(refreshToken)
      if (!refreshed) throw redirect({ to: '/login' })
      setCookie('sb-access-token',  refreshed.accessToken,  { ... })
      setCookie('sb-refresh-token', refreshed.refreshToken, { ... })
      user = refreshed.user
    }

    const tenantId = (user.app_metadata?.tenant_id as string | undefined) ?? ''
    if (!tenantId) throw redirect({ to: '/login?error=no_tenant' })

    // tenants は RLS OFF。dbAdmin (service role) で読み、status をチェック
    const tenantRows = await dbAdmin
      .select({ status: tenants.status })
      .from(tenants).where(eq(tenants.id, tenantId)).limit(1)
    if (!tenantRows[0] || tenantRows[0].status !== 'active') {
      throw redirect({ to: '/login?error=tenant_suspended' })
    }

    return next({ context: { user: { id: user.id, email: user.email ?? '', tenantId, role } } })
  },
)
JWT 検証は Supabase に投げている: 自前で JWKS を取って検証はしていない。 supabase.auth.getUser(token) が Supabase の /auth/v1/user に問い合わせる。 毎リクエスト 1 回ネットワーク往復が発生するが、5 秒タイムアウト (auth.ts) で守ってある。

5. アクセストークンのリフレッシュ

access_token (1h) が切れた瞬間にユーザーがログアウトさせられる UX を避けるため、 ミドルウェアが 透過的に refresh_token を使ってペアを更新する。失敗時のみ /login へ。

1
verifyAccessToken が null を返す (期限切れ or 無効)
↓
2
supabase.auth.refreshSession({ refresh_token })
↓ 新しい access_token + 新しい refresh_token
3
両方の Cookie を setCookie() で上書き (リクエストが終わる前に Set-Cookie ヘッダに乗る)
Supabase の refresh_token は使用ごとにローテーションされる (Refresh Token Rotation)。 古い refresh_token は使えなくなる。盗まれたトークンの再利用検知に役立つが、 現状 fumireply には「盗用検知時にそのユーザーの全セッションを失効させる」フックは無い。

6. テナントコンテキストと RLS

認証で確定した tenantId は、DB 操作のたびに Postgres セッションへ流し込む必要がある。 これを担うのが withTenant()。

app/src/server/db/with-tenant.ts

export async function withTenant<T>(tenantId: string, fn: (tx: TenantTx) => Promise<T>): Promise<T> {
  return db.transaction(async (tx) => {
    await tx.execute(sql`SELECT set_config('app.tenant_id', (${tenantId}::uuid)::text, true)`)
    return fn(tx)
  })
}

set_config(name, value, is_local=true) はそのトランザクション内でのみ効く設定。 RLS ポリシーが tenant_id = current_setting('app.tenant_id')::uuid をチェックするので、 違うテナントの行は SELECT で消え、INSERT/UPDATE は WITH CHECK で弾かれる。

app/src/server/db/migrations/0001_rls.sql (抜粋)

ALTER TABLE "connected_pages" ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON "connected_pages"
  FOR ALL
  USING       (tenant_id = current_setting('app.tenant_id', true)::uuid)
  WITH CHECK  (tenant_id = current_setting('app.tenant_id', true)::uuid);
-- conversations, messages, ai_drafts, deletion_log にも同じポリシー
2 つの DB クライアント:
db 通常の Pooler 接続 (anon ロール) — RLS 強制。アプリの全クエリはこっち。
dbAdmin service role 接続 — RLS バイパス。テナント解決やシステム操作のみ (例: webhook が page_id から tenant_id を引くとき、認証ミドルウェアが tenants を引くとき)。

6.1 全体像

1
ブラウザ sb-access-token cookie
↓ HTTP リクエスト
2
authMiddleware: cookie → JWT 検証 → tenantId 抽出 → context へ
↓
3
サーバー関数本体: withTenant(context.user.tenantId, async (tx) => { ... })
↓ SET LOCAL app.tenant_id
4
Postgres: RLS ポリシーが current_setting('app.tenant_id') と各行の tenant_id を突き合わせる
↓
5
他テナントの行は SELECT に出ず、書き込みも WITH CHECK で拒否

7. ログアウト

supabase.auth.signOut({ scope: 'global' }) で Supabase 側の refresh_token を失効させ、両 Cookie を maxAge:0 でクリア。 scope:'global' なのでそのユーザーの全デバイスのセッションが切れる。
