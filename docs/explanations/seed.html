1. 一言まとめ — 2 つの seed の役割分担

E2E seed ローカル / Playwright 用

Meta API はスタブする前提のテストデータ。Page Token はランダムバイト列でよい。代わりに「会話・受信メッセージ・status='ready' の AI 下書き」まで積んでおき、UI が即時動かせる状態にする。

Review seed 検証環境 / 本番テナント用

本物の Meta Page Access Token を AES-256-GCM で暗号化して投入。テナント (Malbek 想定) と connected_pages 1 行だけ作る。会話などのコンテンツは Meta から実際に流れてくるイベントで埋まる。

観点E2EReview
テナントe2e-tenantmalbek
Supabase Auth ユーザー作る (reviewer@example.com + app_metadata.tenant_id)作らない (運用で別途作成)
Page Tokenランダムバイト 60B (復号されない前提)本物を SSM の master key で暗号化
会話 / 受信メッセージ1 件作成作らない
AI 下書きstatus='ready' で 1 件作らない
Meta Send APIテストでスタブ本物を呼ぶ
SSM アクセス不要必要 (master key 取得のため)

2. 前提 — マイグレーションを先に当てる

seed は INSERT しかしない。テーブル + RLS ポリシーの作成は migration が先。

# 1. スキーマ + RLS ポリシーを作る
npm run db:migrate

# 2-A. ローカル E2E テスト用データ
npm run db:seed:e2e

# 2-B. レビュー / 本番テナント用データ
META_PAGE_ACCESS_TOKEN="EAAB..." \
META_PAGE_ID="1234567890" \
META_PAGE_NAME="Malbek Shop" \
npm run db:seed:review

app/package.json

"db:migrate":     "drizzle-kit migrate",
"db:seed:review": "tsx src/server/db/seed/review.ts",
"db:seed:e2e":    "tsx --env-file-if-exists=.env --env-file-if-exists=.env.local src/server/db/seed/e2e.ts"
どちらの seed も postgres superuser 接続 (DATABASE_URL) を使う。アプリ用の anon ロール接続では RLS に阻まれて他テナントの行を作れない / 見えないため。

3. E2E seed (db:seed:e2e)

3.1 投入する 5 ステップ

1
tenants に (slug='e2e-tenant', status='active') を upsert。onConflictDoNothing() + 既存行 SELECT で冪等
↓ tenantId を取得
2
Supabase Auth に reviewer ユーザーを投入。auth.admin.listUsers() でメール一致を探し、いれば updateUserById、無ければ createUser。
app_metadata: { tenant_id, role: 'reviewer' } を必ず書き込む (これが無いとログイン後に弾かれる)
↓
3
connected_pages に (page_id='e2e-page-0001') を upsert。
page_access_token_encrypted は randomBytes(60) = ダミーの 60 バイト。 形式 (12B IV + 16B tag + N) だけ似せてある。E2E では Meta Send API をスタブするので復号は走らない
↓
4
同 (page_id, customer_psid) の既存 conversation があれば DELETE (cascade で messages / ai_drafts も消える) → 新しい conversations + 受信 messages 1 件を INSERT。
lastInboundAt = now - 5 分 にして 24 時間返信窓内であることを保証
↓
5
ai_drafts に status='ready' を入れる。実際の Anthropic 呼び出しは行わず、固定文 + ダミーのトークン数 / レイテンシを記録。UI の ReplyForm が即時 textarea にプリフィルできるようにするため

3.2 Supabase Auth ユーザー作成のポイント

app/src/server/db/seed/e2e.ts

const supabase = createClient(supabaseUrl, supabaseSecretKey, {
  auth: { persistSession: false, autoRefreshToken: false },
})

const { data: list } = await supabase.auth.admin.listUsers({ perPage: 200 })
const existingUser = list.users.find((u) => u.email === reviewerEmail)

if (existingUser) {
  await supabase.auth.admin.updateUserById(existingUser.id, {
    password: reviewerPassword,
    email_confirm: true,
    app_metadata: {
      ...(existingUser.app_metadata ?? {}),     // 既存の他キーは温存
      tenant_id: tenantId,                      // ← 必ず最新の tenantId を上書き
      role: 'reviewer',
    },
  })
} else {
  await supabase.auth.admin.createUser({
    email: reviewerEmail,
    password: reviewerPassword,
    email_confirm: true,                        // メール認証スキップ
    app_metadata: { tenant_id: tenantId, role: 'reviewer' },
  })
}
app_metadata でなければならない理由: Supabase の JWT クレームに自動で乗るのは app_metadata (サーバー由来のメタ情報) だけ。
もう片方の user_metadata はユーザー自身が更新できるのでセキュリティ境界として不適切。 auth-middleware が読むのも user.app_metadata?.tenant_id。

4. Review seed (db:seed:review)

こちらは「本物の Page Token を 1 つ暗号化して入れる」だけのシンプルな seed。会話やメッセージは作らない (実際の Webhook で埋まる)。

1
必須環境変数チェック (DATABASE_URL, META_PAGE_ACCESS_TOKEN, META_PAGE_ID, META_PAGE_NAME)
↓
2
tenants に (slug='malbek', status='active') を upsert
↓
3
SSM から マスター鍵を取得:
const paramName = `${ssmPathPrefix}/master-encryption-key`
const res = await client.send(new GetParameterCommand({ Name: paramName, WithDecryption: true }))
const masterKey = Buffer.from(res.Parameter.Value, 'base64')
↓
4
encryptToken(pageAccessToken, masterKey) で AES-256-GCM 暗号化:
const iv = randomBytes(12)
const cipher = createCipheriv('aes-256-gcm', masterKey, iv)
const enc = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()])
const tag = cipher.getAuthTag()
return Buffer.concat([iv, tag, enc])  // 12 + 16 + N
↓
5
connected_pages に INSERT:
{ tenantId, pageId, pageName,
  pageAccessTokenEncrypted: encrypted,
  webhookVerifyTokenSsmKey: `${ssmPathPrefix}/meta/webhook-verify-token` }
既存 (page_id ユニーク) があれば onConflictDoNothing でスキップ
Page Token の取得は seed の外: スクリプトは「もう手元に Page Token がある」前提。取得手順は meta-tokens.html § 4 を参照。

5. なぜ service_role 鍵が必要か

E2E seed が SUPABASE_SECRET_KEY (= service_role) を要求する理由:

  • パスワード付きユーザーをサーバー側で作るには admin API 必須。anon キーでできるのは「ユーザーが自分でサインアップ」のみ
  • email_confirm: true でメール検証をスキップしたい (テストのたびにメールリンクを踏むのは現実的でない)
  • app_metadata はサーバー専用フィールド。anon ユーザーは書けない (これがセキュリティ要件)
SUPABASE_SECRET_KEY は本番には絶対にコミットしない: .env.local に置いてリポジトリ外に保つ。 漏洩すると全テナントの全データ・全ユーザーが読み書きされる。アプリの実行時 (Lambda) でも、middleware で tenants テーブルを引く目的でのみ使い、テナントスコープのクエリには使わない設計。

6. 冪等性のしくみ

どちらの seed も「何度走らせても同じ最終状態に収束する」を狙っている。実装パターンは 3 つ:

テーブルパターンキー
tenantsINSERT ... ON CONFLICT DO NOTHING + フォールバック SELECTUNIQUE(slug)
connected_pages同上UNIQUE(page_id)
Supabase auth ユーザーlistUsers → 一致あれば updateUserById、無ければ createUseremail
conversations (E2E のみ)DELETE してから INSERT (cascade で messages / ai_drafts も消す)(page_id, customer_psid)
会話だけ delete-then-insert なのは、E2E では「クリーンな受信状態 + ready な下書き」が初期条件として欲しいから。 upsert にすると古い messages / ai_drafts が残ってしまう。
