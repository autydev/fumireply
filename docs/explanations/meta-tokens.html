1. なぜ複数種類のトークンが必要か

Meta のプラットフォームには「誰が誰の代理で何をするか」の階層がある:

  • App (= fumireply という「アプリ」を Meta に登録した存在)
  • User (= そのアプリを使う Facebook ユーザー、例えばオーナーさん)
  • Page (= ユーザーが管理している Facebook ページ)

Messenger 経由で「ページ X として返信を送る」には、ページの代理人として認可された Page Access Token が必要。 ただしそのトークンを直接取ることはできず、まず User Access Token を経由してページのトークンを発行してもらう、という二段ロケットになる。 さらに、Webhook の真正性 (Meta から本当に来たか) を検証するために App 自身の秘密 (App Secret) も使う。

2. 4 種のトークン早見表

User Access Token

誰が: Facebook ユーザー本人 (オーナー)
用途: Page Access Token を発行してもらうための踏み台。
短命 (~60 分) と長命 (~60 日) があり、短命を fb_exchange_token で長命に交換する。
fumireply 内での扱い: 保存しない。Page Token を取ったら破棄。

Page Access Token

誰が: ページ
用途: Send API POST /me/messages でメッセージを送信。
長命の場合は無期限 (User がアプリを再認可しない / パスワード変更しない限り)。
fumireply 内での扱い: テナント単位で DB に AES-256-GCM 暗号化保存。 connected_pages.page_access_token_encrypted (bytea)。

App Secret

誰が: アプリ自身
用途: Webhook 受信時の X-Hub-Signature-256 検証 (HMAC キー)。 また User Token を長命に交換するときの client_secret パラメータ。
fumireply 内での扱い: SSM SecureString で 1 個だけ保管。 パス例 /fumireply/review/meta/app-secret。

Webhook Verify Token

誰が: 任意の文字列 (こちらが決める)
用途: Webhook サブスクリプション登録時の サーバー所有確認。Meta が GET ?hub.verify_token=… を投げてくるので、保存しているものと一致するか確認して hub.challenge を返す。
fumireply 内での扱い: SSM SecureString で 1 個。パス例 /fumireply/review/meta/webhook-verify-token。 テナントごとには持たず、Meta App ごと (= 全テナント共通)。
5 つ目の小要素: PSID (Page-Scoped ID) — トークンではないが関連する。同じ Facebook ユーザーでもページが違えば違う ID が振られる。 fumireply では会話相手の識別子として conversations.customer_psid に保存している。

3. User Token と Page Token の違い (重要)

観点 User Access Token Page Access Token
主体 Facebook ユーザー個人 Facebook ページ (法人格的な存在)
取得方法 OAuth ダイアログ / Graph API Explorer / Login for Business 取得済みの User Token を使って GET /me/accounts を叩く
fumireply で使う場面 セットアップ時の1 回だけ (Page Token を取り出すための踏み台) 毎回の返信送信で必須 (Send API)
有効期限 短命 ~60 分 / 長命 ~60 日 長命の Page Token は実質無期限
失効する条件 時間経過 / ユーザーがパスワード変更 / 認可解除 ユーザーが認可解除 / パスワード変更 / ページから管理者権限剥奪
fumireply での保存 保存しない (取り出したらすぐ破棄) 暗号化して DB の connected_pages.page_access_token_encrypted
覚え方: User Token は「ページの鍵束を取り出すためのオーナーの本人確認」。Page Token は「鍵束から取り出したページ専用の鍵」。 毎日使うのは鍵 (Page Token) だけで、本人確認 (User Token) は鍵を作る瞬間にしか要らない。

4. Page Access Token を取得する流れ

fumireply MVP では、この取得は 手動オペレーションで行う想定 (UI からの OAuth 連携は未実装)。 取得した Page Token を環境変数で seed スクリプトに渡し、暗号化保存する。

1
Meta for Developers のダッシュボードで Graph API Explorer を開き、短命 User Token を発行 (~60 分有効)
↓ fb_exchange_token
2
GET https://graph.facebook.com/v19.0/oauth/access_token
  ?grant_type=fb_exchange_token
  &client_id=<APP_ID>
  &client_secret=<APP_SECRET>
  &fb_exchange_token=<短命 USER TOKEN>
レスポンスの access_token が長命 User Token (~60 日)
↓ GET /me/accounts
3
GET https://graph.facebook.com/v19.0/me/accounts
  ?access_token=<長命 USER TOKEN>
返ってきた配列の各エントリに access_token (= Page Access Token) と id (= Page ID) が入っている
↓
4
取り出した Page Token と Page ID を seed スクリプトに渡す:
META_PAGE_ACCESS_TOKEN="EAAB..." \
META_PAGE_ID="1234567890" \
META_PAGE_NAME="Malbek Shop" \
npm run db:seed:review
App Secret はサーバー側だけが知っている秘密。長命交換 (fb_exchange_token) のときに必要なので、Graph API Explorer で代用しているうちは手元で扱う必要があるが、自動化する場合はサーバー (Lambda など) から呼び出すのが正しい。

5. 保管方法 — DB 暗号化 vs SSM

シークレット保管先個数理由
Page Access Token DB (connected_pages) 暗号化済み テナント x ページ ごと (1:N) マルチテナント SaaS では新規テナントが増えるたびに SSM パラメータを増やしたくない。DB に置けば INSERT 1 発で済む。
マスター鍵 SSM SecureString システム全体で 1 個 Page Token を AES-256-GCM で暗号化するための 32 バイト鍵。SSM に置く理由は KMS 暗号化 + IAM 制御 + Lambda 起動時にメモリキャッシュできるから。
App Secret SSM SecureString Meta App ごと (= 1 個) テナント横断で同じ値。Webhook 署名検証に必要。
Webhook Verify Token SSM SecureString 1 個 同上。Meta App はシステム全体で 1 つの想定。
R-005 の決定 (research.md): 当初案の「テナントごとに /fumireply/<tenant>/page-access-token という SSM パラメータを切る」は、テナント増加時の運用コストが高く撤回。 代わりに「マスター鍵だけ SSM、本体は DB に暗号化保存」に変更している。

6. 暗号化のしくみ (AES-256-GCM)

Page Token は 32 バイトのマスター鍵で AES-256-GCM 暗号化し、結果の bytea を DB に保存する。 保存フォーマットは固定 12B + 16B + N の連結:

IV
12 B
Auth Tag
16 B
Ciphertext
N B (= plaintext と同サイズ)

app/src/server/services/crypto.ts

const IV_LENGTH = 12
const AUTH_TAG_LENGTH = 16
const ALGORITHM = 'aes-256-gcm'

export function encryptToken(plaintext: string, masterKey: Buffer): Buffer {
  const iv = randomBytes(IV_LENGTH)                      // 毎回ランダム
  const cipher = createCipheriv(ALGORITHM, masterKey, iv)
  const ciphertext = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()])
  const authTag = cipher.getAuthTag()                    // 改竄検知用
  return Buffer.concat([iv, authTag, ciphertext])
}

export function decryptToken(blob: Buffer, masterKey: Buffer): string {
  const iv         = blob.subarray(0, 12)
  const authTag    = blob.subarray(12, 28)
  const ciphertext = blob.subarray(28)
  const decipher = createDecipheriv(ALGORITHM, masterKey, iv)
  decipher.setAuthTag(authTag)                            // 改竄されていれば final() で例外
  return Buffer.concat([decipher.update(ciphertext), decipher.final()]).toString('utf8')
}

マスター鍵の取得

const MASTER_KEY_SSM_PATH = '/fumireply/master-encryption-key'

export async function getMasterKey(): Promise<Buffer> {
  if (masterKeyCache) return masterKeyCache             // Lambda 寿命中はメモリ再利用
  const encoded = await getSsmParameter(MASTER_KEY_SSM_PATH)
  const masterKey = Buffer.from(encoded.trim(), 'base64')
  if (masterKey.length !== 32) throw new Error('...')
  masterKeyCache = masterKey
  return masterKey
}
同じ平文でも暗号文は毎回違う: IV を毎回 randomBytes(12) で振り直すため。 これは GCM の安全性要件 (鍵 x IV の組合せを再利用してはいけない) を満たすために必須。

7. 送信時の復号と利用

オペレーターが「送信」ボタンを押す → sendReplyFn が DB から暗号化バイト列と SSM からマスター鍵を引き、復号して Meta Send API へ。

1
getMasterKey() — SSM から取得 (Lambda メモリキャッシュ済みなら即返却)
↓
2
SELECT page_access_token_encrypted FROM connected_pages (テナントスコープ、RLS 越し)
↓
3
decryptToken(blob, masterKey) → 平文 Page Token
↓ POST /v19.0/me/messages?access_token={token}
4
{ "recipient": { "id": "<PSID>" },
  "messaging_type": "RESPONSE",
  "message": { "text": "..." } }
↓
5
200 OK → data.message_id を messages.meta_message_id に保存。Webhook 経由の echo で send_status='sent' に確定する

app/src/server/services/messenger.ts (要約)

const META_API_BASE = 'https://graph.facebook.com/v19.0'

export async function sendMessengerReply({ pageAccessToken, recipientPsid, messageText }) {
  const body = JSON.stringify({
    recipient: { id: recipientPsid },
    messaging_type: 'RESPONSE',                        // 24h 以内の返信
    message: { text: messageText },
  })

  // 最大 3 回リトライ (500ms / 1500ms / 4500ms)、5xx と timeout のみリトライ対象
  for (let attempt = 0; attempt < 3; attempt++) {
    const response = await fetch(
      `${META_API_BASE}/me/messages?access_token=${pageAccessToken}`,
      { method: 'POST', headers: { 'Content-Type': 'application/json' }, body,
        signal: AbortSignal.timeout(5000) },
    )
    // ... ステータスごとに ok / token_expired / outside_window / permission_denied / ... に分岐
  }
}

8. トークン関連の Meta API エラー

HTTPerror.codeerror_subcodefumireply 上の扱い
400190—token_expired (Page Token 失効 / 取り直し必要)
400102018278outside_window (24h 過ぎ → メッセージタグが必要)
400その他—invalid_request
403——permission_denied
5xx——meta_server_error (リトライ対象)
(timeout)——timeout (1 回までリトライ)
token_expired への対応 (現状): 自動再取得は未実装。エラーをユーザーに見せて、運用者が手動で Page Token を取り直して再 seed する想定。 Phase 2 では Login for Business で UI 側から再認可フローを組む計画 (research.md / plan.md 参照)。
