全体像 — 5 アクター

このアプリの認証は 2 段構えです。

  1. Supabase Auth でアプリ自体のログイン (Email/Password)。Cookie ベース。
  2. ログイン後、各テナントが Facebook Login for Business で Facebook Page を 1 つ連携 (= connected_pages に row が入る) して初めて /inbox 等の業務画面に到達できる。

重要な設計判断: Browser は FB の short-lived user token しか触らない。 Long-lived user token と Page access token は全部サーバー側でしか復号できない暗号化済みで Cookie / DB に保管される。これにより JS の流出経路で page を勝手に subscribe される攻撃を遮断している。

Part 2. 認証付きリクエスト (authMiddleware)

(app) ルートグループ配下のあらゆる server function は app/src/server/middleware/auth-middleware.ts の authMiddleware を通る。これが session の真偽 + tenant の生存確認を行い、 context.user に { id, email, tenantId, role } を載せる。

sequenceDiagram autonumber participant B as Browser
(any server fn call) participant M as authMiddleware participant SA as Supabase Auth participant DB as Postgres
(tenants) participant H as Inner Handler
(例: exchangeAndListFn) B->>M: HTTP request
Cookie: sb-access-token, sb-refresh-token M->>M: getCookie('sb-access-token') alt access token 無し M-->>B: 302 → /login end M->>SA: auth.getUser(accessToken) alt 有効 SA-->>M: user else 失効 M->>SA: auth.refreshSession({ refresh_token }) SA-->>M: 新しい access_token + refresh_token + user M-->>B: Set-Cookie で更新 (1h / 30d) end M->>M: user.app_metadata.tenant_id 抽出 M->>DB: SELECT tenants.status WHERE id = tenant_id alt status != 'active' M-->>B: 302 → /login?error=tenant_suspended end M->>H: next({ context: { user } }) H-->>B: handler の結果
テナント停止 (banned) の運用: レビュワーアカウント (reviewer@malbek.co.jp) は平常時 Supabase 側で banned_until 未来日に設定されている。これは signInWithPassword 段階で弾かれる。一方 tenants.status='suspended' は この middleware が弾く層。「Supabase 個人」と「アプリのテナント」が 2 重 gate になっている。

Part 4. Page 確定 & Webhook 購読 (Step 2)

Step 1 で受け取った Pages 一覧から 1 つを選び「連携」を押した瞬間。 Page access token は Cookie から取り出した long user token を使ってサーバー側で再取得する (browser を信用しない設計)。

用語メモ

  • short-lived user token — FB.login 直後に取れる ~1h 有効のユーザートークン。漏れても被害が限定的
  • long-lived user token — fb_exchange_token で交換した ~60d 有効のユーザートークン。/me/accounts はこれを使う
  • page access token — /me/accounts が返す各 Page 固有のトークン。Send API / subscribed_apps はこれを使う。基本「never expire」(Login for Business 経由なら data_access 期限あり)
  • Login for Business (config_id) — 旧 Facebook Login の後継。permissions / asset types を App Dashboard 側の構成 ID に固定する方式
