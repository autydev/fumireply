# Feature Specification: Draft 操作 UX 強化（再生成・破棄・日本語訳）

**Feature Branch**: `004-draft-actions-and-translation`
**Created**: 2026-05-20
**Status**: Draft
**Input**: User description: "draft 操作 UX を強化する 3 つの機能 (再生成・破棄・日本語訳) を 1 spec にまとめる。spec 003 の実装完了を前提。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Draft を破棄して未対応に戻す (Priority: P1)

オーナーがスレッド詳細画面で生成された AI ドラフトを確認したとき、内容が顧客に合わない／文脈を取り違えていると判断したら、ワンクリックでドラフトを「破棄」できる。破棄するとドラフトは UI から消え、スレッドは「ドラフト未生成」と同等の表示に戻る。

**Why this priority**: 不適切なドラフトをそのまま送ってしまうリスクの軽減が最優先課題。破棄機能がなければオーナーは「送らない・無視する」という消極的回避しかできず、画面ノイズが残り続けて運用が滞る。本機能は最も小さい変更で最大の運用安全性を提供する。

**Independent Test**: ドラフト付きの会話を 1 件用意し、破棄ボタンを押下したのち画面リロードしても再表示されないこと、また同会話を再度開いた他のオーナーにも破棄後の状態が共有されることを確認すれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** スレッド詳細にステータス pending のドラフトが表示されている、**When** オーナーが「破棄」ボタンを押す、**Then** ドラフト表示が消え、スレッドはドラフト未生成相当の状態になる
2. **Given** ドラフトを破棄した直後、**When** 別のブラウザで同じスレッドを開く、**Then** 同じ「ドラフト未生成相当」状態が見える
3. **Given** ドラフトを破棄した会話、**When** 次に新しい顧客メッセージが届く、**Then** 通常のドラフト生成フローが再び動作する
4. **Given** ドラフトを破棄したスレッド、**When** 履歴／管理画面でドラフト履歴を参照する手段が将来追加された場合、**Then** 破棄されたドラフトは内部的に保持されていて参照できる（本 spec では UI 露出なし）

---

### User Story 2 - Draft を再生成して別案を得る (Priority: P2)

オーナーが生成された AI ドラフトを見て「方向性は近いが書き直したい」と感じたとき、「再生成」ボタンを押すと、現在のドラフトはサイレントに古いものとして退避され、同じ会話コンテキストで新しいドラフトが生成・表示される。

**Why this priority**: 破棄に比べて頻度は中程度。破棄してから手動返信を書く運用でも回避は可能だが、AI に任せられる範囲を広げて時短する効果がある。破棄機能の上に成り立つため P2。

**Independent Test**: ドラフト付き会話を 1 件用意し、再生成ボタンを押下した結果、ドラフト本文が変化し、かつ旧ドラフトが内部的に履歴として残っていることを確認する。

**Acceptance Scenarios**:

1. **Given** スレッド詳細にステータス pending のドラフトが表示されている、**When** オーナーが「再生成」ボタンを押す、**Then** 旧ドラフトは「superseded」扱いとなり、新しいドラフトが同じ場所に表示される
2. **Given** 再生成中、**When** 新しいドラフトの生成が完了するまで数秒かかる、**Then** ボタン押下から表示完了までの間、ユーザーは「生成中」のフィードバックを受け取り、二重押下できない
3. **Given** 再生成を 3 回連続で行った会話、**When** 内部 DB を確認する、**Then** 同じ会話に対し 4 つのドラフトレコード（pending 1 + superseded 3）が保持されている
4. **Given** 翻訳設定 OFF の状態で再生成、**When** 新しいドラフトを生成、**Then** 翻訳結果は NULL のまま保存される
5. **Given** 翻訳設定 ON の状態で再生成、**When** 新しいドラフトを生成、**Then** 新ドラフトに対しても翻訳が実行されて保存される

### User Story 3 - Draft の日本語訳を内部確認する (Priority: P3)

英語メッセージ中心の運用において、オーナーが AI 生成ドラフトの意味を素早く把握したい場合、Settings ページで「ドラフトの日本語訳を表示する」をオンにできる。オンにすると、以降生成されるドラフトには日本語訳が付与され、スレッド詳細にドラフト本文と並んで（または折りたたみで）表示される。

**Why this priority**: 英語が読めるオーナーには不要、運用ペースが上がってから欲しくなる補助機能。なくても運用は回るため P3。ただし設定 1 つで切り替えられる体験は提供する。

**Independent Test**: Settings で翻訳を ON にし、新規メッセージ受信 → ドラフト生成のフローを実行、ドラフト本文と並んで日本語訳が表示されることを確認する。OFF に戻すと、以降の新規ドラフトには訳がつかないこと、過去の翻訳済みドラフトも UI に訳が表示されないことを確認する。

**Acceptance Scenarios**:

1. **Given** Settings で翻訳が OFF（デフォルト）、**When** 新規ドラフトが生成される、**Then** 翻訳結果は保存されず、UI にも訳は出ない
2. **Given** Settings で翻訳が ON、**When** 新規ドラフトが生成される、**Then** ドラフト本文と一緒に日本語訳がスレッド詳細に表示される
3. **Given** 翻訳 ON 状態でドラフトが生成された後、Settings で OFF に切り替え、**When** そのスレッドを開く、**Then** 既存ドラフトの訳は UI 表示されない（DB には残っているが非表示）
4. **Given** 翻訳 ON 状態、**When** 何らかの理由で翻訳の取得に失敗（API 制限到達／ネットワークエラー）、**Then** ドラフト本文は通常通り表示され、訳の欄には「取得失敗」と表示される。ドラフトの送信可否には影響しない
5. **Given** 既に破棄されたドラフト、**When** スレッド詳細を再表示、**Then** 破棄ドラフトもその翻訳も UI には表示されない

---

### Edge Cases

- **破棄後すぐに新規メッセージ**: 顧客が直後にメッセージを送ってきた場合、新メッセージに対する新規ドラフト生成が通常通り動く。破棄したドラフトのコンテキストを引きずらない
- **再生成の連打**: 短時間に複数回再生成を押された場合、最後のリクエストだけが pending として残り、途中の生成は superseded か破棄される。少なくとも UI に「pending」が複数並ぶことはない
- **再生成中の破棄**: 再生成リクエストが進行中（旧ドラフトは superseded 化済、新ドラフトはまだ未到着）に破棄ボタンが押された場合、到着した新ドラフトは即座に discarded とする。中間状態のドラフトを UI に滞留させない
- **翻訳 ON 中の DeepL Free 月次上限到達**: 月 50 万文字を超えた以降のドラフトは翻訳取得失敗となる。ドラフト本体の生成は影響を受けず、訳欄には失敗表示が出る。Settings 画面で「今月の翻訳が制限に達しています」を簡易表示できると望ましい
- **翻訳 OFF で過去ドラフトに翻訳が残っている**: DB の翻訳列に値があっても UI 上は表示しない。「保存されているが非表示」状態は許容
- **同じ会話で連続して 10 回以上再生成された場合**: DB 上では全て履歴として保持。UI が重くなる懸念はあるが、UI には常に最新の pending 1 件しか表示しないため問題ない
- **複数オーナー運用での同時操作**: A が破棄を押した瞬間に B が再生成を押した場合、最後に DB に着いた操作が勝つ（一貫性は楽観的）。レアケースなので明示的なロックは設けない

## Requirements *(mandatory)*

### Functional Requirements

#### 破棄機能

- **FR-001**: システムは、スレッド詳細画面にステータス pending のドラフトが存在するとき、「破棄」ボタンを表示しなければならない
- **FR-002**: 破棄ボタン押下時、システムは該当ドラフトのステータスを「discarded」に変更し、UI 上のドラフト表示を消さなければならない
- **FR-003**: システムは、破棄されたドラフトを DB から物理削除してはならない（将来のフィードバック分析のため履歴として保持）
- **FR-004**: 破棄後、同じ会話への新着メッセージに対するドラフト自動生成フローは通常通り動作しなければならない

#### 再生成機能

- **FR-005**: システムは、スレッド詳細画面にステータス pending のドラフトが存在するとき、「再生成」ボタンを表示しなければならない
- **FR-006**: 再生成ボタン押下時、システムは現在の pending ドラフトのステータスを「superseded」に変更し、同じ会話に対して新しいドラフトを生成しなければならない
- **FR-007**: 再生成されたドラフトは、spec 003 で確立済みのドラフト生成パイプライン（顧客固有 custom_prompt、会話コンテキスト、要約 context）を使用しなければならない
- **FR-008**: 再生成中、ユーザーは「生成中」を示す UI フィードバックを受け取り、再度ボタンを押下できない状態でなければならない
- **FR-009**: 同一会話で pending 状態のドラフトが同時に複数存在してはならない（最終的に最後の 1 件のみ pending）
- **FR-010**: superseded ドラフトは DB に保持され、UI には表示されないが、内部参照やデータ分析からはアクセスできなければならない

#### 翻訳機能

- **FR-011**: Settings ページに「ドラフトの日本語訳を表示する」グローバルトグルを追加しなければならない。デフォルトは OFF
- **FR-012**: トグルが ON の状態で新規ドラフトが生成されたとき、システムは DeepL Free 翻訳サービスでドラフト本文を日本語に翻訳し、結果をドラフトレコードに保存しなければならない
- **FR-013**: トグルが OFF の状態では、翻訳サービスを呼び出してはならない（コスト・帯域節約）
- **FR-014**: 再生成時、トグル ON ならば新しいドラフトに対して翻訳を再取得しなければならない（旧ドラフトの翻訳は流用しない）
- **FR-015**: 翻訳取得に失敗した場合（API 上限・ネットワークエラー等）、ドラフト本文の表示・送信機能には影響を与えず、翻訳欄には「取得失敗」を示す UI を表示しなければならない
- **FR-016**: トグルが OFF の状態で、過去に翻訳済みのドラフトを表示するとき、翻訳テキストは UI に表示してはならない
- **FR-017**: 翻訳 API キーは環境変数として 1 つのみ管理され、すべてのオーナー・ページ・会話で共有される

#### 共通

- **FR-018**: 本機能で追加されるすべての UI 要素・テキストは i18n（日英）対応をしなければならない
- **FR-019**: すべての DB スキーマ変更は既存の `drafts` テーブルへの列追加で完結し、新規テーブルを作成してはならない
- **FR-020**: 本機能の実装は、spec 003 のマージ後に開始される前提とし、spec 003 で導入された `drafts` 関連テーブル・カラム構成に依存してよい

### Key Entities *(include if feature involves data)*

- **Draft**: AI が生成した返信ドラフト。`status` フィールドが `pending` / `discarded` / `superseded` のいずれかを取る。新たに `translation_ja`（日本語訳本文、NULL 可）と `translation_status`（翻訳の取得結果状態）の概念を持つ
- **Settings (Owner-scoped)**: spec 003 で導入された設定オブジェクト。新たに「ドラフト日本語訳の有効/無効」フラグを 1 つ追加する。デフォルト OFF
- **Translation Result**: ドラフトに紐づく日本語訳のテキストと取得状態。失敗時もドラフト本体には影響しない

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ドラフトを破棄したいと判断したオーナーは、ドラフト表示から 1 アクション（ボタン押下のみ・確認ダイアログなし、または 1 段階のみ）で破棄を完了でき、所要時間は 3 秒以内に収まる
- **SC-002**: 再生成ボタンを押してから新ドラフトが画面に表示されるまでの体感時間は、95 パーセンタイルで 10 秒以内に収まる（spec 003 の生成パイプラインに律速されるため、その性能特性に従う）
- **SC-003**: 翻訳機能を ON にして 30 日運用したとき、運用継続中に DeepL Free の月次上限（50 万文字）に到達してドラフト本体生成が止まる事故がゼロ件（翻訳失敗があっても本体は動く設計が満たされていること）
- **SC-004**: オーナー満足度調査（または自己申告）で、「不適切なドラフトをそのまま見続ける」という運用ストレスが本機能リリース後に 80% 以上で減少したと回答される
- **SC-005**: 再生成・破棄を含む全てのドラフト履歴が DB に保持され、後日「あるオーナーが直近 30 日に破棄／再生成した回数」を SQL クエリ 1 本で集計できる

## Assumptions

- spec 003（顧客コンテキストと設定階層）が本 spec 着手前にマージ済みであり、`drafts` テーブルと Settings ページが運用に乗っている
- ドラフト生成パイプラインは spec 003 で完成済みで、本 spec はそれを再利用するだけ。AI ワーカー側の大規模改修は不要
- 翻訳サービスは DeepL Free を使用する（無料枠 50 万文字/月、API キーは個人または小規模チーム単位で 1 つ）。将来的に DeepL Pro や別サービスに切り替える場合の抽象化は本 spec では実装しない
- 顧客送信用の本文は常に元のドラフト本文（翻訳前）であり、翻訳は内部表示専用。顧客に和訳を送信する経路は本 spec のスコープ外
- 破棄理由収集 UI、編集距離計測、フィードバックに基づく自動プロンプト改善などは本 spec のスコープ外（将来の spec で扱う）
- 再生成・破棄のオペレーションはオーナーロール（ページに対する書き込み権限を持つユーザー）のみが行える。読み取り専用ユーザーには操作 UI を出さない
- DB のドラフト履歴保持には件数制限を設けない。将来必要になったときにデータ保持ポリシーを別 spec で定める
- 翻訳設定はオーナースコープのグローバル設定であり、ページ単位／会話単位の上書きは本 spec では提供しない
