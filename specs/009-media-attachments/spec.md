# Feature Specification: 受信画像・添付メディアの永続保存とスレッド表示

**Feature Branch**: `009-media-attachments`
**Created**: 2026-07-18
**Status**: Draft
**Input**: User description: 顧客が Messenger で送ってきた画像・動画・音声・ファイル添付を、Webhook 受信時に即ダウンロードして永続保存し (Meta の CDN URL は短命なため)、メッセージに添付情報を保持し、スレッド UI で画像表示・種別ラベル表示する。過去の期限切れメディアはプレースホルダ表示。送信側 (fumireply からの画像送信) はスコープ外。親 Issue: autydev/fumireply#73

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 顧客が送った画像がスレッドで画像として表示される (Priority: P1)

顧客は問い合わせの際に、商品の写真やスクリーンショットなどの画像を Messenger で送ってくることがある。現状の fumireply では、その画像はスレッド上に生の URL 文字列として表示されるだけで、しかもその URL は数日で失効するため後から開くこともできない。本ストーリーでは、受信した画像を受信時点で fumireply 側に永続保存し、スレッド上で**画像そのもの**として表示できるようにする。

**Why this priority**: 画像を確認できないと運営者は問い合わせ内容を理解できず、返信のために Meta のインボックスへ移動せざるを得ない。fumireply を一次窓口として成立させるうえで最も利用頻度が高く影響の大きい欠落のため P1。

**Independent Test**: 顧客役のアカウントから Messenger で画像を 1 枚送信し、fumireply のスレッド画面でその画像がサムネイル表示されること、クリックで原寸表示できること、および数日後 (Meta 側 URL の失効後) に再度スレッドを開いても表示できることを確認できる。

**Acceptance Scenarios**:

1. **Given** 顧客との会話が存在する、**When** 顧客が Messenger で画像を送信する、**Then** その画像は受信時に永続保存され、スレッド上に画像として表示される (生の URL 文字列は表示されない)
2. **Given** 画像を含むメッセージがスレッドに表示されている、**When** 運営者がその画像をクリックする、**Then** 原寸の画像が表示される
3. **Given** 受信から数日が経過し Meta 側の配信 URL が失効している、**When** 運営者がスレッドを開く、**Then** 画像は fumireply が保存したコピーから引き続き表示される
4. **Given** 顧客が 1 通のメッセージで複数の画像を送信した、**When** そのメッセージを受信する、**Then** すべての画像が保存され、スレッド上にすべて表示される

---

### User Story 2 - 画像以外の添付も種別が分かる形で表示される (Priority: P2)

顧客は画像以外にも動画・音声 (ボイスメッセージ)・ファイル・スタンプを送ってくることがある。現状はこれらの種別判定自体が存在せず (動画・音声・ファイルは「不明」に落ちる)、スレッド上は空のバブルとして表示されるため、運営者は「何かが送られてきた」ことすら気づけない。本ストーリーでは、これらの添付の種別を正しく判定・記録し、スレッド上で「動画」「音声」「ファイル」などの種別ラベルとして表示して空バブルをなくす。

**Why this priority**: 画像ほど頻度は高くないが、空バブルは「顧客の発言を見落とす」という運用上の実害に直結する。インライン再生までは求めず、まず何が送られてきたかを可視化する。

**Independent Test**: 顧客役のアカウントから動画・音声・ファイル・スタンプをそれぞれ送信し、fumireply のスレッド上でそれぞれが空バブルではなく種別の分かる表示になっていることを確認できる。

**Acceptance Scenarios**:

1. **Given** 顧客との会話が存在する、**When** 顧客が動画・音声・ファイルのいずれかを送信する、**Then** そのメッセージは正しい種別として記録され、スレッド上に種別の分かるラベルで表示される (空バブルにならない)
2. **Given** 顧客がスタンプを送信した、**When** そのメッセージを受信する、**Then** スレッド上にスタンプであることが分かる表示が出る
3. **Given** 種別を判定できない未知の添付が届いた、**When** そのメッセージを受信する、**Then** 「不明な添付」であることが分かる表示になり、空バブルにはならない

---

### User Story 3 - 取得できないメディアはプレースホルダで明示される (Priority: P2)

メディアの保存は必ず成功するとは限らない (受信時のダウンロード失敗、サイズ超過など)。また、本機能リリース以前に受信済みのメディアは Meta 側 URL がすでに失効しており復元できない。本ストーリーでは、表示できないメディアを空バブルや壊れた画像アイコンではなく、「画像 (取得不可)」のような明示的なプレースホルダとして表示する。

**Why this priority**: User Story 1 の品質を支える受け皿。失敗やレガシーデータを黙って隠すと運営者が会話の文脈を誤解するため、「メディアがあったが表示できない」ことを正直に伝える必要がある。

**Independent Test**: 本機能リリース前に受信済みの画像メッセージを含むスレッドを開き、生 URL や壊れた画像ではなくプレースホルダが表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 受信時のメディア保存が失敗したメッセージがある、**When** 運営者がスレッドを開く、**Then** そのメッセージは種別付きのプレースホルダ (例: 「画像 (取得不可)」) として表示される
2. **Given** 本機能リリース以前に受信した画像メッセージ (メディア未保存・URL 失効) がある、**When** 運営者がスレッドを開く、**Then** 生の URL 文字列ではなくプレースホルダが表示される
3. **Given** メディア保存が失敗した、**When** その受信処理が完了する、**Then** メッセージ自体はスレッドに記録されており、失敗が運用ログから確認できる

---

### Edge Cases

- 1 通のメッセージに複数の添付が含まれる場合 — すべての添付を順序を保って記録・保存し、スレッド上にすべて表示する (現状は先頭 1 件しか見ていない)
- 添付のダウンロードが失敗した場合 (URL 失効・ネットワークエラーなど) — メッセージの取り込み自体は成功させ、該当添付は「取得不可」として記録する。失敗は構造化ログに残す
- 添付のサイズが上限を超える場合 — 保存をスキップし、種別の記録は維持してプレースホルダ表示とする。スキップは構造化ログに残す
- 外部送信 (echo) に添付が含まれる場合 — inbound と同様に保存を試みる。少なくとも種別の記録は必ず維持し、空バブルにしない (006 の「echo 非テキストは本文空」方針のうち、種別記録は維持しつつ表示を改善する)
- テキストと添付が同一メッセージに共存する場合 — 本文と添付の両方を表示する
- スタンプ — 種別表示のみとし、スタンプ画像自体の保存は行わない (Meta 提供の定型画像であり顧客固有データではないため)
- 過去 (本機能リリース前) に受信済みでメディア未保存のメッセージ — 復元手段がないためプレースホルダ表示で対応する。遡及取得は行わない
- 本文に生の CDN URL が保存されている過去の画像メッセージ — URL 文字列をそのまま表示せず、プレースホルダに置き換えて表示する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは受信メッセージの添付種別として、画像・スタンプに加えて動画・音声・ファイルを判定できる。判定できないものは「不明」として記録する
- **FR-002**: システムは inbound メッセージの添付メディア (画像・動画・音声・ファイル) を、受信時に即ダウンロードして自社管理の永続ストレージに保存する (Meta の配信 URL は短命で、受信直後が唯一確実な取得タイミングであるため)
- **FR-003**: メディアの保存に失敗しても、メッセージ自体の取り込み (スレッドへの記録) は必ず成功させる。保存失敗は添付単位で記録し、構造化ログに残す
- **FR-004**: メッセージ本文に Meta の配信 URL を保存する現行動作を廃止する。非テキストメッセージの本文は空とし、添付情報は本文とは別に保持する
- **FR-005**: システムは 1 メッセージあたり複数の添付を、種別・保存先参照・元メッセージ内での順序とともに保持できる
- **FR-006**: スレッド UI は、保存済みの画像添付を画像として表示し、クリックで原寸表示できる
- **FR-007**: スレッド UI は、メディアが取得できない添付 (保存失敗・本機能以前の受信データ) を種別付きプレースホルダとして表示する
- **FR-008**: スレッド UI は、スタンプ・動画・音声・ファイル・不明の各種別を、空バブルではなく種別の分かる表示にする
- **FR-009**: 外部送信 (echo) 経路の添付についても保存を試み、保存可否に関わらず種別の記録は維持する
- **FR-010**: 保存されたメディアは顧客の個人データとして扱い、テナント間で分離する。保存先の構造とアクセス制御の両面で、他テナントのメディアへ到達できないことを保証する
- **FR-011**: 保存されたメディアへのアクセスは、認証済みかつ該当テナントに属する運営者に限定する。認可を伴わない恒久公開 URL でメディアを配信しない
- **FR-012**: 1 添付あたりの保存サイズ上限を設ける。上限超過時は保存をスキップし、種別記録とプレースホルダ表示、構造化ログでの記録を行う
- **FR-013**: メディアの保存成功・保存失敗 (ダウンロード失敗/サイズ超過) を区別できる構造化ログイベントを既存の命名規約に沿って出力し、CloudWatch Logs Insights で件数を集計可能にする。カスタムメトリクスやアラートは追加しない
- **FR-014**: 添付の取り込み・表示は、既存の AI 下書き生成・未返信バッチ判定の挙動を変えない (非テキストメッセージの扱いは従来と同等)

### Key Entities *(include if feature involves data)*

- **メッセージ (Message)**: 既存エンティティ。本文・種別に加えて、0 個以上の添付情報を持つようになる
- **添付 (Attachment)**: 1 メッセージに属する 1 つの添付を表す。種別 (画像/動画/音声/ファイル/スタンプ/不明)、メッセージ内での順序、保存済みメディアへの参照 (保存失敗時は参照なし = 取得不可) を持つ
- **保存済みメディア (Stored Media)**: fumireply が自社管理ストレージに保存した添付の実体。テナント・会話・元メッセージに紐づくキー構造で格納され、テナント分離の単位となる

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 本機能リリース後に受信した画像メッセージは、受信直後だけでなく Meta 側 URL の失効後 (受信から 7 日以上経過後) もスレッド上で画像として表示できる
- **SC-002**: スレッド上に生の配信 URL 文字列が表示されるメッセージが 0 件になる (過去データ含む)
- **SC-003**: スレッド上の空バブル (種別も内容も分からないメッセージ) が 0 件になる。すべての非テキストメッセージは画像・種別ラベル・プレースホルダのいずれかで表示される
- **SC-004**: 添付メディアの保存失敗が発生しても、メッセージ自体の取り込み成功率は 100% を維持する (添付起因のメッセージ欠落 0 件)
- **SC-005**: 通常サイズの添付について、受信時のメディア保存成功率が 99% 以上である (CloudWatch Logs Insights の成功/失敗イベント集計で測定)
- **SC-006**: あるテナントの運営者が別テナントの保存済みメディアへ到達できたケースが 0 件である
- **SC-007**: 運営者が受信画像の内容確認のために Meta 側インボックスへ移動する必要がなくなる (リリース後、画像確認起因の Meta インボックス併用が解消)

## Assumptions

- メディアの保存先は自社管理のオブジェクトストレージとする。ストレージや配信方式の具体的な選定は計画フェーズで決定する
- 保存したメディアの保持期間は、当面「メッセージと同寿命 (明示的な削除なし)」とする。会話履歴が参照可能な限りメディアも表示できるべきであり、削除だけ先行するとプレースホルダが増えて UX を損なうため。コスト最適化のためのライフサイクル (例: 低頻度アクセス層への移行) は運用開始後のストレージコスト実績を見て別途判断する
- 1 添付あたりのサイズ上限の具体値は、受信処理基盤の制約 (メモリ・タイムアウト) と Messenger 側の送信上限 (約 25MB) を踏まえて計画フェーズで決定する
- 動画・音声のインライン再生、ファイルのダウンロード提供は本機能のスコープ外とする (種別ラベル表示まで)。将来、保存済みメディアを使って拡張できる余地は残す
- 送信側 (fumireply から顧客へ画像等を送る) 機能はスコープ外とする
- 本機能リリース以前に受信済みで Meta 側 URL が失効したメディアの復元は行わない (Meta から再取得する手段がないため)。プレースホルダ表示で対応する
- スタンプは Meta 提供の定型画像であり、保存対象としない (種別表示のみ)
- Webhook の購読フィールド追加は不要 (添付情報は既存の `messages` 購読イベントに含まれる)。Meta App 管理画面での追加作業は発生しない前提
- 006 で定めた「echo の非テキストは本文空 + 種別記録」の方針は、本機能で「種別記録 + 添付保存 (可能な場合)」に更新される。関連 spec 群 (006 spec.md / research.md) の該当記述は本機能の実装時に同期する
