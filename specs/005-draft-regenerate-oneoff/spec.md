# Feature Specification: AI 下書きの条件付き再生成 (ワンオフ指示)

**Feature Branch**: `005-draft-regenerate-oneoff`
**Created**: 2026-06-23
**Status**: Draft
**Input**: User description: "AI 下書きカードに『再生成』ボタンとワンオフ指示入力欄を追加し、運営者が会話のアクティブ下書きを 1 回限りの追加条件付きで作り直せるようにする (GitHub Issue #66)。"

## Clarifications

### Session 2026-06-23

- Q: ワンオフ指示の上限文字数 → A: 1000 文字 (UI 上にカウンタ表示を必須化)
- Q: 再生成失敗時のアクティブ下書きの扱い → A: 旧本文を保持したまま `ready` に戻し、カードに一過性のエラーメッセージを表示。ワンオフ指示は入力欄に保持して再試行可能 (新状態の追加なし)
- Q: 再生成中に同一会話へ新着 Messenger メッセージが届いた場合 → A: 進行中の再生成ジョブを優先し、新着到来時の自動バッチ再生成は抑制。再生成完了後に未取り込みの新着が残っていれば従来通り自動再生成 (ワンオフ指示は適用されない) が走る
- Q: 再生成ジョブのタイムアウト方針 → A: クライアント側ポーリングが 90 秒経っても完了しない場合、FR-011 と同じ失敗扱い (旧本文保持 + 一過性エラー + ボタン再活性化 + 指示は入力欄に保持) とする

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ワンオフ指示付きで下書きを作り直す (Priority: P1)

運営者が顧客から価格や在庫などのその場限りの情報を求められたとき、自分で値を確認したうえで「OP-09 は ¥800、これで案内」のような追加条件を 1 回だけ AI に渡し、会話の AI 下書きを作り直す。指示は会話に永続化されず、その回の生成にのみ影響する。

**Why this priority**: 本機能のコアバリュー。現状 AI 下書きは新着メッセージ受信時に自動生成されるのみで、運営者が「この 1 回だけの条件」を AI に渡す手段が存在しない。これがないと、ワンオフ情報を含む返信は手書きするしかなく、AI 下書きの活用率が大きく下がる。

**Independent Test**: アクティブ下書きがある会話を開き、再生成入力に任意の指示を入力して再生成を実行する。完了後に下書き本文が指示を反映した内容で上書きされており、再度ページを開き直しても会話側にその指示が残っていないことで検証可能。

**Acceptance Scenarios**:

1. **Given** 会話に `ready` 状態のアクティブ下書きが 1 件ある, **When** 運営者がワンオフ指示「OP-09 は ¥800、これで案内」を入力して再生成する, **Then** 同じ会話のアクティブ下書きが `pending` を経て `ready` に戻り、本文に「¥800」が反映され、会話を再度開いてもワンオフ指示欄は空に戻っている。
2. **Given** ワンオフ指示と会話の永続 custom_prompt が両方ある, **When** 再生成する, **Then** ワンオフ指示が永続 custom_prompt より優先的に反映される (例: 永続側「丁寧調」+ ワンオフ「カジュアル調で」なら出力はカジュアル調)。
3. **Given** 再生成を実行した直後, **When** ページを再読み込みする, **Then** ワンオフ指示の入力内容は復元されず空欄になっている (会話への保存は行われない)。

---

### User Story 2 - 指示なしで素の再生成をする (Priority: P2)

運営者が「内容は気に入らないが特に追加条件はない、もう一度生成してほしい」と感じたとき、ワンオフ指示を空のまま再生成ボタンを押す。AI は最新の会話履歴と既存設定 (会話の tone_preset / custom_prompt / 顧客プロファイル等) のみで再度下書きを作る。

**Why this priority**: 「単純に作り直したい」は頻度の高いユースケース。実装も P1 と同経路で `instruction` を空にすればよく、追加コストが小さい。P1 がなくても素の再生成だけで一定の価値はあるが、P1 と比べると体験差分が小さいので P2。

**Independent Test**: アクティブ下書きがある会話で、ワンオフ指示を空のまま再生成ボタンを押す。下書きが `pending` を経て `ready` に戻り、内容が再生成 (前回と同一・類似でも可) されていることで検証可能。

**Acceptance Scenarios**:

1. **Given** アクティブ下書きが `ready`, **When** ワンオフ指示を空のまま再生成する, **Then** 下書きが `pending` を経て `ready` に戻り、新しい本文で上書きされる (旧本文は残らない)。

---

### User Story 3 - 再生成中の状態と完了の可視化 (Priority: P2)

運営者が再生成ボタンを押した瞬間から完了までの間、AI 下書きカードが「生成中」であることを示し、完了したら自動的に最新本文に切り替わる。運営者は別の作業に移っていても、戻ってきたときに状態が分かる。

**Why this priority**: ボタンを押した後の進捗が見えないと「押せたのか」「失敗したのか」が分からず、誤って何度も押す/手書きに切り替えてしまうリスクがある。P1 の体験を成立させるために必要だが、機能としては既存の下書きステータス表示 (#004) の再利用で成立するため P2。

**Independent Test**: 再生成を実行した直後にカードがローディング/pending 表示になり、生成完了後に手動操作なしで `ready` 状態と新本文に切り替わることで検証可能。

**Acceptance Scenarios**:

1. **Given** アクティブ下書きが `ready`, **When** 再生成を実行する, **Then** カードは即時に pending 表示に切り替わる。
2. **Given** バックエンドで再生成が完了した, **When** 運営者がそのまま画面に留まっている, **Then** ページ再読み込みなしで `ready` 表示と新本文に切り替わる。
3. **Given** 再生成中, **When** 運営者がもう一度再生成ボタンを押そうとする, **Then** ボタンは再生成完了まで非活性化されており、二重発火しない。

---

### Edge Cases

- 再生成対象のアクティブ下書きが存在しない会話 (まだ AI 下書きが作られたことがない / 既に送信済み) で再生成を押した場合は、機能を提供しないかボタン自体を出さないことで防ぐ。
- ワンオフ指示の入力が極端に長い (例: 数千〜数万文字) 場合は、ユーザーに分かる形で上限を伝えるか、超過分を許容しない。プロンプト爆発と外部 API コスト暴騰を避けるための上限を持つ。
- 再生成中に同じ会話で新着 Messenger メッセージが届いた場合、進行中ジョブを優先して走らせ、新着到来時の自動バッチ再生成は抑制する。再生成完了後に未取り込みの新着がまだ残っていれば、従来通り自動再生成 (ワンオフ指示なし) が走って最新メッセージを反映した下書きで上書きする。結果として運営者は最終的に「最新メッセージを反映した 1 件の下書き」を得るが、ワンオフ指示が次のサイクルにまたぐことはない。
- 再生成ジョブが AI 側エラー等で失敗した場合、対象下書きは旧本文を保持したまま `ready` に戻り、カードに失敗を示す一過性メッセージを表示する。ワンオフ指示の入力は維持され、運営者はそのまま再試行できる。
- ネットワーク断などで再生成リクエスト自体が送れなかった場合、ワンオフ指示の入力内容は失われずに残り、再試行できる。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 運営者は、アクティブ下書きがある会話画面の AI 下書きカードから「再生成」操作を起動できる MUST。
- **FR-002**: 「再生成」操作は、その回限りで使い捨てるワンオフ指示テキストの入力欄を伴う MUST。入力欄は空のまま実行することも許容する。
- **FR-003**: ワンオフ指示は、当該会話に永続保存してはならない MUST NOT。再生成完了後やページ再読み込み後に入力欄は空に戻る。
- **FR-004**: 再生成を実行すると、対象会話のアクティブ下書きは即座に「生成中」状態へ遷移し、完了時に最新の AI 出力で本文が上書きされる MUST。旧バージョンは保持しない。
- **FR-005**: ワンオフ指示が与えられた場合、AI 出力は会話の永続 custom_prompt や未返信メッセージのバッチ要件よりもワンオフ指示を優先して反映する MUST。
- **FR-006**: ワンオフ指示が空の場合は、既存の会話・顧客・ページ設定だけで再生成し、結果はやはり「最新で上書き」される MUST。
- **FR-007**: 「生成中」状態は AI 下書きカードに視覚的に表示され、ユーザーがページに留まっている限り完了時に自動的に「準備完了」状態と新本文に切り替わる MUST。
- **FR-008**: 再生成リクエストが進行中の間、同一会話の「再生成」操作は二重発火を防ぐために抑制される MUST。同様に、再生成ジョブが進行中の間は新着 Messenger メッセージ到来による自動バッチ再生成も抑制し、進行中ジョブ完了後に未取り込み新着があれば改めて自動再生成を実行する。
- **FR-009**: アクティブ下書きが存在しない会話、または下書きが既に確定/送信済みの状態にある会話では「再生成」操作は提示されない、または操作不能である MUST。
- **FR-010**: ワンオフ指示テキストの上限は 1000 文字 (UTF-16 コードユニットでカウント) とし、入力欄には残り文字数カウンタを常時表示し、上限超過時は再生成ボタンを非活性化して上限超過を運営者へ明示する MUST。
- **FR-011**: 再生成処理が失敗した場合、対象アクティブ下書きは旧本文を保持したまま `ready` 状態へ戻し、AI 下書きカードに失敗を示す一過性のエラーメッセージを表示する MUST。ワンオフ指示の入力内容は入力欄に維持し、運営者はそのまま再試行できる。失敗で会話のアクティブ下書きが永久に「生成中」のまま残ってはならない。クライアントは再生成開始から 90 秒経過しても `ready` 復帰が観測できない場合、本要件の失敗扱いと同じ UX (旧本文保持・エラー表示・ボタン再活性化・指示の入力欄保持) を発火する MUST。
- **FR-012**: 再生成は #004 で導入されたアクティブ下書き 1 件モデルを破壊しない MUST。新規の下書きレコードを追加で増やすのではなく、既存のアクティブ下書きを作り直す経路として動作する。

### Key Entities *(include if feature involves data)*

- **アクティブ下書き (AI Draft, active)**: 会話に対して常に最大 1 件存在する AI 生成下書き。状態 (生成中 / 準備完了など) と本文を持つ。再生成はこのエンティティの状態と本文を更新する操作で、新規エンティティは作らない。
- **ワンオフ指示 (Operator Instruction, one-off)**: 運営者がその 1 回の再生成のためだけに与える追加プロンプト断片。会話・顧客・ページのどのエンティティにも永続化されず、再生成ジョブのライフサイクル内でのみ存在する一過性データ。
- **再生成ジョブ (Regeneration Job)**: 「アクティブ下書きを最新条件 (＋ワンオフ指示) で作り直す」非同期処理単位。会話への紐付け、ワンオフ指示有無、開始/完了/失敗の状態を持つ。完了すると対象アクティブ下書きが更新される。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 運営者は、AI 下書きカードを開いた状態から、ワンオフ指示を入力して再生成を開始するまでを 15 秒以内で完了できる。
- **SC-002**: 通常負荷下で、再生成ボタンを押してから新しい下書き本文がカード上に表示されるまでの体感時間が 30 秒以内に収まる (P50)。
- **SC-003**: ワンオフ指示で「価格」「在庫」「日付」などの具体値を渡したケースのうち、生成結果本文にその具体値がそのまま含まれる割合が 95% 以上。
- **SC-004**: 再生成完了後にページを再読み込みした際、ワンオフ指示が会話側に残存しているケースが 0 件 (回帰テストで担保)。
- **SC-005**: 再生成導線の導入後 1 か月以内に、AI 下書きをそのまま送信せず手書きで全文置き換えるケースの割合が、導入前比で 20% 以上減少する。

## Assumptions

- #004 で確立された「会話スコープのアクティブ下書き 1 件」モデルが本番で稼働しており、再生成はこのモデルの上に被せる前提とする。バージョン履歴は持たず、最新の生成結果で上書きする。
- ワンオフ指示は会話・顧客・ページのいずれにも永続保存しない。永続的に効かせたい指示は既存の `conversations.custom_prompt` (003 で導入された DraftSettingsEditor) を使う運用とする。
- 再生成は同期 API ではなく非同期ジョブ経由で行い、生成中状態と完了通知は #004 のアクティブ下書きステータス機構 (pending → ready 等) を流用する。
- 同一会話に対する再生成ジョブの同時走行は禁止し、進行中は新規発火を抑制する (UI 側のボタン抑制 + バックエンド側の状態チェックの二段)。
- 認証・権限は既存の運営者用セッションをそのまま使い、対象会話のページに対する権限を持つ運営者のみが再生成できる。
- 失敗時のエラーメッセージや上限文字数の具体値は、実装時に運用上安全な既定値を定義する (本仕様では「上限が存在する」「失敗を認識できる」までを規定)。
