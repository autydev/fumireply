# Feature Specification: AI 下書き生成のクラッシュ修正と失敗時の確実な状態反映

**Feature Branch**: `008-fix-ai-worker-timestamp`
**Created**: 2026-07-18
**Status**: Draft
**Input**: User description: "GitHub issue #75: ai-worker が max(timestamp) の型不一致 TypeError でクラッシュし再生成/AI下書きが失敗するバグの修正。ai-worker/src/handler.ts の未返信バッチ境界取得で生 SQL `max()` が string を返し、PgTimestamp.mapToDriverValue が TypeError → Lambda クラッシュ → SQS 3 リトライ後 DLQ 行き、ai_drafts が pending のまま放置される。必須修正は型付きカラム select + orderBy desc + limit 1 への置き換え。併せて堅牢化として (1) processDraftJob 全体の try/catch で pending 取り残し防止、(2) Lambda timeout 60s とリトライラダー最悪 133s の不整合解消、(3) DLQ redrive の運用手順を検討。"

## 背景(不具合の概要)

2026-06-27(feature 004 バッチ下書き導入)以降、AI 下書きの生成・再生成が「返信済みメッセージが 1 件でも存在する会話」で必ず失敗している。生成ジョブが AI 呼び出し前に内部エラーで異常終了し、3 回の自動再試行をすべて使い切って失敗キューに退避される。下書きレコードは「生成中」のまま放置され、オペレーターの画面では 90 秒(自動生成は 60 秒)のポーリング後にタイムアウトエラーが表示されるか、下書きバナーが黙って消える。

根本原因は、未返信バッチ境界(最後の送信メッセージ時刻)の取得処理が日時値を誤った型で受け取り、後続の比較処理が実行時エラーを起こすこと。返信が 1 件もない新規会話のみフォールバック値で成功するため、「よく失敗する」という症状として観測されていた。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 返信済み会話でも AI 下書きが生成できる (Priority: P1)

オペレーターとして、過去に返信したことのある会話(=実運用のほぼすべての会話)で、AI 下書きの自動生成・手動再生成・ワンオフ指示つき再生成が正常に完了してほしい。

**Why this priority**: 本不具合の根本修正。現状、返信済みの実会話ではほぼ 100% 失敗しており、AI 下書き機能が実質使えていない。

**Independent Test**: 送信メッセージを 1 件以上含む会話で再生成を実行し、タイムアウトせずに下書きが表示されることで単独検証できる。

**Acceptance Scenarios**:

1. **Given** 送信(outbound)メッセージが 1 件以上ある会話, **When** オペレーターが AI 下書きの再生成を実行する, **Then** 下書きが正常に生成され、タイムアウトエラーが表示されない
2. **Given** 送信メッセージが 1 件以上ある会話に顧客から新着メッセージが届いた, **When** 自動バッチ下書きが起動する, **Then** 下書きが正常に生成されて画面に表示される
3. **Given** 送信メッセージが 1 件もない新規会話, **When** 下書き生成が起動する, **Then** 従来どおり全受信メッセージを未返信として下書きが生成される(既存挙動の退行なし)
4. **Given** 最後の送信メッセージ以降に受信メッセージがある会話, **When** 下書きが生成される, **Then** 未返信の範囲(最後の送信以降の受信メッセージ)が正しく下書きの対象になる

---

### User Story 2 - 生成に失敗したら「生成中」のまま放置されない (Priority: P2)

オペレーターとして、下書き生成が何らかの理由で失敗した場合には、「生成中」のままポーリングのタイムアウトを待たされるのではなく、失敗が状態として確実に記録され、画面に速やかに反映されてほしい。

**Why this priority**: 今回の障害で被害を拡大させたのは「失敗が状態に書き込まれず pending のまま放置される」構造。根本原因を直しても、想定外の失敗(DB エラー等)で同じ体験の劣化が再発しうる。

**Independent Test**: 生成処理の途中(AI 呼び出しの前後いずれか)で意図的に例外を発生させ、下書きが失敗状態(再生成の場合は従来下書き保持+エラー通知)に遷移することで単独検証できる。

**Acceptance Scenarios**:

1. **Given** 自動バッチ生成のジョブ処理中に予期しない内部エラーが発生した, **When** ジョブが終了する, **Then** 下書きは「失敗」状態として記録され、「生成中」のまま残らない
2. **Given** 手動再生成のジョブ処理中に予期しない内部エラーが発生した, **When** ジョブが終了する, **Then** 従来の下書き本文は保持されたままエラーが記録され、オペレーターに失敗が通知される(005 の失敗時挙動と同一)
3. **Given** ジョブが失敗状態を書き込んで終了した, **When** クライアントが次にポーリングする, **Then** タイムアウトを待たずに失敗が画面に反映される

---

### User Story 3 - AI 応答が遅くても処理が途中で打ち切られない (Priority: P3)

オペレーターとして、AI サービスの応答が遅延・不安定なときでも、再試行を含む生成処理が実行環境の時間制限で強制打ち切りされ「生成中」のまま放置される、という事態を避けたい。

**Why this priority**: 現状、AI 呼び出しの再試行戦略の最悪所要時間(約 133 秒)が処理基盤の実行時間上限(60 秒)を超えており、AI 側の遅延時に User Story 2 と同じ「pending 放置」が別経路で再発する。発生頻度は低いが構造的な不整合。

**Independent Test**: 再試行戦略の最悪所要時間と実行時間上限を机上で突き合わせ、上限内に収まる(または上限が引き上げられている)ことを構成値で検証できる。

**Acceptance Scenarios**:

1. **Given** AI 呼び出しが再試行を最大回数まで繰り返す最悪ケース, **When** 処理全体の所要時間を見積もる, **Then** 実行環境の時間制限内に収まり、途中で強制終了されない
2. **Given** 実行時間上限に関連する待機設定(キューの可視性タイムアウト等)がある, **When** 上限を変更する, **Then** 関連設定も整合するよう更新されている

---

### User Story 4 - 障害期間中に失敗したジョブの後始末 (Priority: P4)

運用者として、修正の本番反映後に、障害期間中(6/27 以降)に失敗キューへ退避されたジョブをどう扱うか(再投入するか破棄するか)の手順を明確にし、必要なら実施したい。

**Why this priority**: ユーザー影響は将来の生成には及ばないが、未返信のまま放置されている会話の下書きを回復できる可能性がある。運用作業であり、コード修正から独立して実施できる。

**Independent Test**: 手順書に従って失敗キューの滞留ジョブを再投入し、返信済み会話のジョブは副作用なく終了、未返信会話のジョブは下書きが生成されることで検証できる。

**Acceptance Scenarios**:

1. **Given** 修正が本番反映済みで失敗キューに障害期間中のジョブが滞留している, **When** 手順書に従って再投入する, **Then** すでに返信済みの会話のジョブは下書きを作らず終了し、未返信の会話には現時点の内容に基づく下書きが生成される
2. **Given** 再投入を実施しない判断をした場合, **When** 保持期限(14 日)が経過する, **Then** ジョブは自動的に消滅し、その旨が手順書に明記されている

---

### Edge Cases

- 送信メッセージが 1 件もない会話(フォールバック境界)で従来挙動が維持されるか
- 最後の送信メッセージと同時刻の受信メッセージがある場合、未返信境界の判定が修正前後で変わらないか(境界は「より後」の受信のみを未返信とする従来仕様を維持)
- 失敗状態の書き込み自体が失敗した場合(DB 接続断など)、ジョブの再試行によって最終的に状態が書き込まれるか、少なくとも従来より悪化しないか
- 再生成の失敗時挙動(従来下書き保持+エラー通知)と自動生成の失敗時挙動(失敗状態)の区別が、予期しない例外経由の失敗でも守られるか
- 失敗キューの再投入時、対象の会話・下書きレコードがすでに削除/更新されている場合に副作用なく終了するか

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、送信メッセージの有無にかかわらず、すべての会話で AI 下書きの自動生成・手動再生成・ワンオフ指示つき再生成を完了できなければならない(送信メッセージ存在時の実行時エラーを解消する)
- **FR-002**: 未返信バッチの境界判定(最後の送信メッセージ以降の受信を未返信とする)は、修正前の仕様と同一の結果を返さなければならない(境界仕様の変更はしない)
- **FR-003**: 下書き生成ジョブが予期しない内部エラーで失敗した場合、システムは下書きを「生成中」のまま放置せず、失敗を状態として記録しなければならない。失敗時の状態遷移は既存仕様に従う: 自動生成は「失敗」、再生成は「従来下書き保持+エラー記録」
- **FR-004**: AI 呼び出しの再試行戦略の最悪所要時間は、処理基盤の実行時間上限の範囲内に収まらなければならない(再試行戦略の短縮または上限の引き上げのいずれかで整合させ、関連する待機設定も同時に整合させる)
- **FR-005**: 送信メッセージが存在する会話での下書き生成・再生成を対象とした自動回帰テストを追加しなければならない
- **FR-006**: 障害期間中に失敗キューへ退避されたジョブの扱い(再投入手順と、再投入しない場合の自然消滅)を運用手順として文書化しなければならない
- **FR-007**: ジョブの失敗と回復の状況を運用者が確認できるよう、失敗時には原因を特定できる構造化ログが出力されなければならない(既存のログ規約に従う)

### Key Entities

- **AI 下書き (ai_drafts)**: 会話ごとのアクティブな下書き。状態は「生成中(pending)」「利用可能(ready)」「失敗(failed)」等。今回の不具合では pending のまま放置される
- **メッセージ (messages)**: 会話内の受信(inbound)/送信(outbound)メッセージ。送信メッセージの最終時刻が未返信バッチの境界
- **生成ジョブ**: キュー経由で処理される下書き生成の作業単位。失敗時は最大 3 回試行後に失敗キューへ退避される

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 送信メッセージを含む会話での下書き再生成が、タイムアウトエラーなしで完了する(修正前: ほぼ 100% 失敗 → 修正後: 内部エラー起因の失敗 0 件)
- **SC-002**: 修正反映後、下書き生成の処理系での実行時エラー(異常終了)の発生が 0 件になる
- **SC-003**: 修正反映後、失敗キューへの新規流入が 0 件になる(AI サービス側の長期障害を除く)
- **SC-004**: 生成に失敗した下書きが「生成中」のままクライアントのポーリング上限(90 秒)を超えて残ることがない
- **SC-005**: 送信メッセージあり/なし両方の会話パターンが自動回帰テストで検証されている

## Assumptions

- 未返信バッチ境界の仕様(最後の送信メッセージより後の受信メッセージを未返信とみなす)自体は正しく、変更しない。修正は境界値の取得方法の不具合解消のみ
- 再生成失敗時の「従来下書き保持+エラー通知」(005 で確立)と自動生成失敗時の「失敗状態」という既存の失敗時挙動を、予期しない例外経路にもそのまま適用する
- 失敗キューの滞留ジョブは、処理時点の最新の会話状態を再読みして動作するため、再投入しても古い内容の下書きが作られたり二重の副作用が生じたりしない(返信済み会話では下書きを作らず終了する)。再投入の実施可否は運用判断とし、本 feature では手順の文書化までを必須とする
- 実行時間上限と再試行戦略の整合方法(上限引き上げ or 再試行短縮)は計画フェーズで決定する。いずれの場合もオペレーターの体験(クライアントのポーリング上限 90 秒以内に結果が反映されること)を優先して選択する
- DB スキーマの変更は不要
