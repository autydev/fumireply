# Feature Specification: MVP for Meta App Review Submission

**Feature Branch**: `001-mvp-app-review`
**Created**: 2026-04-20
**Updated**: 2026-04-30 (architecture pivot: Supabase + Anthropic API、AI 下書き生成を MVP コア機能に昇格)
**Status**: Draft
**Input**: User description: "この会話の流れでMVPで申請をだすところまでのアプリと、messenger-cs-tool-design-v2.md と mockをみて、仕様を検討して下さい。"

## Overview

Meta App Review（Facebook Developers のアプリ審査）を通過するために必要な、**Messenger 受信 → AI 下書き生成 → 人間が確認・編集して送信**という Human-in-the-Loop ワークフローを最小実装した審査用バージョンの仕様。Meta レビュワーがホスティングされた管理画面にログインし、実際に Facebook ページ宛てに送られたメッセージを受信・返信する一連のフローを再現できることが目的。

本仕様は `messenger-cs-tool-design-v2.md` の Phase 1 MVP から **顧客管理・商品管理・Slack 通知・Instagram DM・AI 自動分類**を除外し、**Messenger 単一チャネルでの「受信 → AI 下書き生成 → 人間が確認/編集 → 送信」**に絞り込んだ審査通過用サブセットである。AI 下書き生成は Claude Haiku 4.5（Anthropic API）を SQS Worker Lambda から非同期に呼ぶ。送信は管理画面のボタン押下による完全に人間トリガで行う（自動送信はしない）。

審査通過後、同一 App ID 配下で本番機能（AI による自動分類、顧客管理、Instagram DM 等）を段階的に追加していく前提。申請時の Use Case 説明文に含めた機能範囲内であれば、追加実装時の再審査は原則不要となる。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Meta レビュワーが管理画面にログインして疎通確認できる (Priority: P1)

Meta レビュワーが申請フォームに記載されたテスト用アカウントと URL を使い、ホスティングされた管理画面にログインする。ログイン後、連携済みの Facebook ページに届いたメッセージが受信一覧に表示されていることを確認できる。

**Why this priority**: 管理画面へのログインと受信メッセージ表示ができないと、レビュワーはそもそもアプリの動作を確認できず、審査を通せない。これが審査通過の最小かつ絶対条件。

**Independent Test**: レビュワー用テストアカウントでログイン → 事前に Messenger 経由で送信されたテストメッセージが一覧に表示されることを確認。これ単体で「アプリが稼働していて Webhook が機能している」ことが実証される。

**Acceptance Scenarios**:

1. **Given** 公開済みの管理画面 URL と発行済みのテスト用ログイン情報がある、**When** レビュワーが URL にアクセスしログイン情報を入力する、**Then** 10 秒以内に受信一覧画面が表示される
2. **Given** レビュワーがログイン済み、**When** テスト用 Facebook アカウントから連携ページに Messenger メッセージを送信する、**Then** 30 秒以内に受信一覧にそのメッセージが表示される
3. **Given** 管理画面が審査期間中に稼働している、**When** レビュワーが任意のタイミングでアクセスする、**Then** 画面が 10 秒以内に応答し、受信済みメッセージを閲覧できる

---

### User Story 2 - Meta レビュワーが AI 下書きを確認・編集して送信できる (Priority: P1)

レビュワーは受信一覧から特定の会話を開き、過去のやり取りを時系列で確認する。スレッド画面には**受信メッセージに対して AI が事前生成した返信下書き**が表示されており、レビュワーは下書きをそのまま採用して送信するか、編集してから送信するか、空の状態から自由入力で送信するかを選べる。「送信」ボタンを押すと Send API が呼ばれ、Messenger 側でメッセージが実際に届いていることを確認できる。

**Why this priority**: `pages_messaging` 権限の利用と、AI 下書き生成という本アプリの中核価値の両方を実証する中核フロー。送信操作が成立しなければ申請対象の権限を「実際に使っている」証拠が示せず、AI 下書きが表示されなければ Use Case 説明文の "AI-assisted reply drafting" の根拠を欠く。P1 とする。

**Independent Test**: 受信済みメッセージに対してスレッドを開く → AI 下書きが表示されている → そのまま送信、もしくは編集して送信 → Messenger アプリで返信が届くことを確認。Webhook 受信、AI 下書き生成、Send API 送信の 3 端が動くことが単独で実証される。

**Acceptance Scenarios**:

1. **Given** 受信一覧に 24 時間以内に受信したメッセージがある、**When** レビュワーがそのスレッドを開く、**Then** 受信内容に応じた AI 生成の返信下書きが返信欄にプレースホルダとして表示されている（生成中の場合は「下書き生成中…」を表示し、完了後に置き換える）
2. **Given** AI 下書きが表示されたスレッド、**When** レビュワーが下書きをそのまま、または編集して送信ボタンを押す、**Then** 5 秒以内に送信成功のフィードバックが表示され、Messenger 側のテストアカウントに返信が届く
3. **Given** AI 下書きを使わず空の状態から自由入力したい、**When** レビュワーが下書きを削除して任意の文章を入力し送信する、**Then** 入力した文章が Messenger 側に届く
4. **Given** メッセージ送信に失敗した（トークン期限切れ等）、**When** 管理画面がエラーを受け取る、**Then** レビュワーに失敗理由が分かるエラーメッセージが表示される
5. **Given** AI 下書き生成に失敗した（API 障害等）、**When** スレッドを開く、**Then** 下書きは表示されず空の入力欄になり、レビュワーは自由入力で送信できる

### User Story 3 - 審査に必要な公開ページ群にアクセスできる (Priority: P1)

レビュワーおよび Meta の審査システムは、独自ドメイン上に公開された以下のページを HTTPS 経由で参照できる。

- プライバシーポリシー（**AI 利用に伴う第三者提供（Anthropic）の明示を含む**）
- 利用規約
- ユーザーデータ削除手順
- 会社情報ページ（Business Verification 整合用）

**Why this priority**: これらの URL は App 設定画面の必須入力項目であり、いずれかが 404 / HTTPS 不備 / コンテンツ不足の場合は申請フォーム提出自体がブロックされる、もしくは自動リジェクトされる。AI 下書き生成のために Anthropic にメッセージ本文を送信する事実をプライバシーポリシーで明示しないと審査差し戻しのリスクがある。

**Independent Test**: 各 URL を外部ブラウザから直接開き、HTTPS 証明書が有効で、各ページに必要事項（取得データ、保存期間、第三者提供（Anthropic を含む）、削除窓口、連絡先）が記載されていることを確認。

**Acceptance Scenarios**:

1. **Given** プライバシーポリシー URL が App 設定に登録されている、**When** 任意のブラウザから当該 URL を開く、**Then** HTTPS で配信され、取得項目・保存期間・**第三者提供先（Anthropic に AI 下書き生成のために送信する旨）**・削除窓口・連絡先が記載された日本語または英語のページが表示される
2. **Given** ユーザーデータ削除 URL が登録されている、**When** 当該 URL を開く、**Then** データ削除リクエストの手順または削除コールバック仕様に準拠したレスポンスが返る
3. **Given** 会社情報ページが独自ドメインに公開されている、**When** Business Verification 書類に記載された法人名・住所・電話番号と突合する、**Then** サイト上の記載内容と書類の記載内容が一致する

### User Story 4 - スクリーンキャスト動画で用途を実演できる (Priority: P1)

申請者（Malbek 側）は、上記 Story 1〜2 のフローを画面収録し、各権限（`pages_messaging`、`pages_manage_metadata`、`pages_read_engagement`）がどの画面のどの操作で使われているか、および AI 下書き生成がどのタイミングで行われ人間が必ず最終承認する流れになっているかを、ナレーションまたは字幕で説明した動画を準備できる。

**Why this priority**: スクリーンキャスト動画は App Review 提出フォームで必須アップロード項目であり、動画の品質が審査結果を大きく左右する。「AI が自動送信しているのではなく、人間が確認して送信している」が一目で伝わる動画でないと「自動 bot」と疑われ差し戻されるリスクがある。

**Independent Test**: 動画単独で「どの画面」「何の操作」「どの権限」「ユーザーへの価値」「AI 下書きと人間の承認の関係」が明示されていることを第三者レビューで確認。

**Acceptance Scenarios**:

1. **Given** Story 1〜2 が実機で動作している、**When** 一連のフローを画面収録する、**Then** 受信 → 下書き表示 → 確認 → 編集（任意）→ 送信 → Messenger 到達までが連続した動画として記録される
2. **Given** 動画が存在する、**When** 各シーンに字幕またはナレーションを付加する、**Then** 各権限が「どの画面のどの操作で使われているか」が明示されており、「AI は下書き候補のみを生成し、送信は必ず人間がボタンを押す」旨が明記されている

---

### User Story 5 - 申請通過後に本番機能を追加しても再審査不要な用途記述になっている (Priority: P2)

申請時の Use Case 説明文は、Phase 2 以降に追加予定の機能（AI 自動分類カテゴリ、顧客管理、商品管理、Instagram DM、Slack 通知、請求リンク生成、追跡番号送信）を将来の実装として「同じ権限スコープ内で行う」ことを明示的にカバーする文言で書かれている。AI 下書き生成は MVP に含むため Use Case 説明文の主機能として書く。

**Why this priority**: 初回申請の Use Case 説明に含めておけば、後続機能の追加時に再申請・再審査が不要になる。Messenger 受信/返信の用途範囲内で拡張するかぎり、スコープ追加扱いにならない。審査通過自体には影響しないが、開発効率を大きく左右するため P2 とする。

**Independent Test**: Use Case 説明文をドラフト段階で第三者レビュー（Meta 申請代行サービス情報やコミュニティ情報）に照らして、Phase 2〜3 の機能が同じ権限スコープで読み取れる記述になっていることを確認。

**Acceptance Scenarios**:

1. **Given** Use Case 説明文ドラフトが存在する、**When** Phase 2 の機能リスト（AI 自動分類、顧客管理、商品管理、Slack 通知）を参照する、**Then** それらが Messenger 受信/返信の文脈内で読み取れる文言になっている
2. **Given** Use Case 説明文が提出される、**When** 申請が承認される、**Then** Phase 2 機能を追加した際の再審査が不要となる

---

### Edge Cases

- Webhook Callback URL が審査期間中に一時的にダウンした場合：Meta 側は疎通失敗を検知し「Cannot reproduce」で差し戻す可能性がある。24/7 稼働を維持するホスティングプランが必須。
- Page Access Token の有効期限切れ：開発用短期トークンは約 60 分で失効する。審査中は長期トークン（Never-expiring Page Token）に差し替えておく必要がある。
- 24 時間窓の超過：レビュワーが受信から 24 時間以上経過してから返信しようとした場合、`pages_messaging` だけでは送信できない。審査中はレビュワー側にメッセージ受信直後の返信を促すか、テスト用のインバウンドメッセージを定期的に入れておく運用が必要。
- レビュワー用テストアカウントの 2FA：2FA が有効だとレビュワーがログインできない。IP 制限や 2FA は審査中は無効化しておく。
- 同一メッセージの重複表示：Meta Webhook はまれに同一イベントを複数回配信する。受信一覧上で重複が起きないように Meta のメッセージ ID での冪等処理が必要。
- 空メッセージ／スタンプのみ／画像のみの受信：テキスト以外のメッセージタイプを受信したとき、管理画面が落ちずにプレースホルダ表示できる。テキスト以外は AI 下書き生成の対象外とし、空の入力欄を表示する。
- 同時ログイン：レビュワーが複数のセッションで同時ログインした場合でも、両方のセッションから閲覧・送信ができる。
- AI 下書き生成失敗：Anthropic API 障害・タイムアウト・レート制限等で下書き生成が失敗した場合、画面は空の入力欄を表示しレビュワーは自由入力で送信できる。下書き生成失敗が送信フローを止めないこと。
- Supabase 無料枠の自動 Pause：無料プランは 7 日間アクティビティがないと DB が自動 Pause される。審査期間中に Pause されると Webhook 全失敗 → 審査差し戻しの致命的リスク。**EventBridge Scheduled Rule で 1 日 1 回 keep-alive クエリを発行し、Lambda 内部で 3 回リトライ + 最終失敗時に SNS 通知 + 「36 時間 Invocation なし」アラーム + OnFailure Destination の多重防御で確実に Pause を回避する運用を必須とする。**1 日 1 回にすることで、1 度の失敗があっても残り 6 日間の対応猶予がある。

## Requirements *(mandatory)*

### Functional Requirements

#### 受信・表示
- **FR-001**: システムは Facebook ページに宛てて送信された Messenger メッセージを Webhook 経由で受信し、受信後 30 秒以内に管理画面の受信一覧に表示しなければならない。
- **FR-002**: 管理画面の受信一覧は、顧客ごとに会話スレッドを集約し、時系列で最新順に表示しなければならない。
- **FR-003**: システムは同一メッセージ ID の重複受信を検知し、管理画面上で重複表示を防がなければならない。
- **FR-004**: システムは Meta からの Webhook 署名（X-Hub-Signature-256）を検証し、署名不一致のリクエストは処理せず 401 を返さなければならない。

#### AI 下書き生成（MVP コア機能）
- **FR-022**: システムは inbound メッセージを受信した時点で、Worker Lambda から Anthropic API（Claude Haiku 4.5）を呼び出し、当該メッセージへの返信下書きを非同期で生成しなければならない。
- **FR-023**: 生成された下書きは `ai_drafts` として永続化され、スレッド画面で受信メッセージに紐づいて表示されなければならない。
- **FR-024**: 下書き生成は受信から **60 秒以内**に完了することを目標とし、未完了時は画面に「下書き生成中…」を表示しなければならない。
- **FR-025**: 下書き生成が失敗した場合（API 障害、タイムアウト、レート制限等）、スレッド画面は空の入力欄を表示しレビュワーは自由入力で送信できなければならない。
- **FR-026**: AI 下書きは**自動送信されてはならない**。送信は必ず管理画面の送信ボタン押下によって行われる（Human-in-the-Loop 必須）。

#### 返信送信
- **FR-005**: ユーザー（レビュワー／オペレーター）は受信したメッセージのスレッドを開き、AI 下書きをそのまま、編集、または空入力から自由入力して送信できなければならない。
- **FR-006**: システムは送信操作を受けて 5 秒以内に Meta Send API を呼び出し、結果（成功／失敗）をユーザーにフィードバックしなければならない。
- **FR-007**: 送信に成功したメッセージは会話スレッド内の送信済みメッセージとして即座に表示されなければならない。
- **FR-008**: 送信に失敗した場合、ユーザーは失敗理由（トークン期限切れ／24 時間窓超過／ネットワーク等）を画面上で確認できなければならない。

#### 認証・アクセス制御
- **FR-009**: 管理画面にアクセスするユーザーは、ログイン認証（Supabase Auth、メールアドレス＋パスワード）を通過しなければならない。
- **FR-010**: システムはレビュワー用テストアカウントを最低 1 件発行でき、そのアカウントは IP 制限・2FA を課さず任意の環境からログインできなければならない。
- **FR-011**: 管理画面の全ページ、API、Webhook エンドポイントは HTTPS でのみ配信されなければならない。

#### 公開ページ群
- **FR-012**: システムはプライバシーポリシーページを独自ドメイン上に公開し、以下を明記しなければならない：取得するデータ項目、利用目的、保存期間、**第三者提供先（AI 下書き生成のために Anthropic にメッセージ本文を送信する旨を含む）**、ユーザーによる削除依頼窓口、問い合わせ連絡先。
- **FR-013**: システムは利用規約ページを独自ドメイン上に公開しなければならない。
- **FR-014**: システムはユーザーデータ削除のエンドポイントまたは手順ページを公開し、Meta のデータ削除コールバック仕様に準拠したレスポンスを返すか、手動削除依頼の手順を掲載しなければならない。削除対象には `ai_drafts` 行も含めなければならない。
- **FR-015**: システムは会社情報ページ（法人名・住所・電話番号）を独自ドメイン上に公開し、Business Verification 書類との整合を取らなければならない。

#### 稼働・運用
- **FR-016**: 管理画面、Webhook エンドポイント、公開ページ群は、審査期間中（申請提出から承認／差し戻しまで）24 時間 365 日稼働し続けなければならない。
- **FR-017**: Webhook エンドポイントは Meta からの受信リクエストに対し 20 秒以内に HTTP 200 を返さなければならない。受信 Lambda は署名検証 → DB INSERT → SQS enqueue → 200 のみを行い、AI 下書き生成は Worker Lambda に分離する。
- **FR-018**: システムは Page Access Token の有効期限切れを検知した場合、管理画面上で管理者に通知しなければならない。
- **FR-027**: システムは Supabase 無料プランの自動 Pause（7 日無アクティブで発動）を回避するため、EventBridge Scheduled Rule で **1 日 1 回**以上 DB に keep-alive クエリ（軽量 SELECT 1 等）を発行する keep-alive Lambda を稼働させなければならない。Lambda は内部で指数バックオフによる 3 回リトライを実装し、最終失敗時は SNS 経由で即時通知を発報しなければならない。1 日 1 回 + 7 日の Pause 閾値により、失敗が連続しても運用者には複数日の対応猶予がある。

#### 申請ドキュメント
- **FR-019**: 申請者はスクリーンキャスト動画を準備し、Story 1〜2 のフローと権限利用箇所、および「AI 下書きは候補に過ぎず送信は人間が承認する」旨を動画内で明示できなければならない。
- **FR-020**: 申請者は Use Case 説明文を準備し、AI 下書き生成を主機能として記述しつつ、Phase 2 以降に想定される機能拡張が同じ権限スコープ内で読み取れる文言にしなければならない。
- **FR-021**: 申請者はテスト用 Facebook ページを 1 つ用意し、レビュワーが当該ページに Messenger メッセージを送れる状態にしなければならない。

### Key Entities *(include if feature involves data)*

- **Tenant**: SaaS の契約事業者単位。テナント名、slug、プラン（`free`/`pro`/`enterprise`、MVP は `free` 固定）、ステータス、課金 ID（Phase 2）を保持する。MVP では 1 件（Malbek）を seed。
- **Connected Page**: 連携済みの Facebook ページ情報。tenant_id、ページ ID、ページ名、AES-256-GCM 暗号化された Page Access Token、連携日時を保持する。1 tenant に複数ページ可能（MVP は 1 tenant × 1 page）。
- **Conversation**: 特定の顧客（Messenger ユーザー）と当該ページの会話スレッド。顧客 PSID、最終受信日時、最終送信日時、24 時間窓の残時間目安を保持する。
- **Message**: 個別のメッセージ。方向（受信／送信）、本文、送信者ハンドル、タイムスタンプ、Meta メッセージ ID（冪等キー）、送信ステータス（送信済／失敗／未送信）を保持する。
- **AI Draft**: 受信メッセージに紐づいて Worker Lambda が生成した返信下書き。生成元 message ID、下書き本文、生成ステータス（pending/ready/failed）、使用モデル、生成所要時間を保持する。
- **Admin User**: 管理画面にログインするユーザー。Supabase Auth 側に保存。所属テナントは `user_metadata.tenant_id` で表現（1 user = 1 tenant）。アプリ DB には Supabase Auth UUID の参照のみ保持し、メールアドレスやパスワードは持たない。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Meta App Review 提出後、初回または 2 回目の審査サイクル以内に必要権限（`pages_messaging`、`pages_manage_metadata`、`pages_read_engagement`）が Advanced Access で承認される。
- **SC-002**: レビュワーが管理画面にログインしてから最初の受信メッセージ一覧を閲覧可能になるまで、10 秒以内で完了する。
- **SC-003**: 顧客から Messenger メッセージが送信されてから管理画面の受信一覧に反映されるまで、30 秒以内で完了する（95 パーセンタイル）。
- **SC-004**: 管理画面での送信操作から Messenger 側の顧客端末にメッセージが届くまで、5 秒以内で完了する（95 パーセンタイル）。
- **SC-005**: 審査期間中（提出から結果通知まで）の管理画面、Webhook、公開ページ群の累積稼働率が 99.5% 以上を維持する。
- **SC-006**: Meta App Review 提出フォームで必須入力とされる 5 つの URL（管理画面、プライバシーポリシー、利用規約、データ削除、Webhook Callback）がすべて HTTPS で 200 を返す。
- **SC-007**: 申請通過後 Phase 2 機能を追加実装したとき、再審査申請を要せず Messenger 受信/返信用途の範囲内で機能追加が完了できる。
- **SC-008**: AI 下書きが受信から 60 秒以内にスレッド画面に表示される（95 パーセンタイル、生成成功時）。

## Assumptions

- **Business Verification は並行して申請する**：本仕様のアプリ構築と、Meta Business Verification（法人書類・会社サイト・電話認証）の申請は並行タスクとして進める。Verification は本仕様には含まれないが、Advanced Access 取得の前提条件である。
- **Messenger 単一チャネル**：Instagram DM（`instagram_manage_messages` 権限）は本 MVP スコープから除外し、将来の別申請で扱う。
- **AI 下書き生成は Anthropic API 直接利用**：AWS Bedrock 経由ではなく Anthropic API（Claude Haiku 4.5）を直接呼ぶ。Lambda は VPC 外配置のため外向き HTTPS が問題なく出せる。
- **AI 自動分類・タグ付けは含まない**：受信メッセージのカテゴリ分類（price/intent/detail/shipping/stock/other）は Phase 2 で追加。MVP の AI は「返信下書きの生成のみ」。
- **顧客管理・商品管理 UI は最小化**：design v2 の顧客 CRUD、VIP タグ、購入履歴、商品マスタ等は審査用 MVP では非提供。受信・スレッド・返信送信に関係しない画面は実装しない。
- **Slack 通知は含まない**：運用補助機能であり、審査には不要。
- **24 時間窓外送信は対象外**：`pages_messaging_subscriptions` 権限（24 時間窓外での送信）は本 MVP では申請しない。審査中は 24 時間以内の受信への返信のみで権限利用を実証する。
- **マルチテナント SaaS 前提（最初から組み込み）**：本アプリは将来の SaaS 販売（月額課金）を見据え、**最初からマルチテナントアーキテクチャ**で構築する。`tenants` テーブル + 全データテーブルの `tenant_id` カラム + RLS（Row Level Security）でテナント間データ分離を強制する。
  - **MVP 期間中は tenant 1 件（Malbek）を seed で事前作成**し、Meta レビュワーは既存テナントにログインする。レビュワーから見れば普通のシングルテナントアプリと変わらない。
  - **セルフサインアップ画面と Stripe 課金統合は Phase 2** で実装する。`tenants.plan` / `tenants.stripe_customer_id` カラムは MVP 時点でスキーマに用意するが、MVP は `plan='free'` 固定。
  - **マルチオペレーター運用 / RBAC は当面実装しない**：1 tenant あたりログインユーザーは 1〜2 名（運営者本人 + 補助）を想定。Supabase Auth の `user_metadata.tenant_id` で 1 user = 1 tenant を表現し、ロール分岐は実装しない。Phase 3 以降で複数オペレーター運用が要求される段階まで保留。
  - **Page Access Token は DB に AES-256-GCM 暗号化して保存**（テナント増加時に SSM 操作が要らない設計）。マスター鍵は SSM `/fumireply/master-encryption-key`。
- **独自ドメインでのホスティング**：会社サイト、管理画面、Webhook、公開ページ群はすべて Malbek 所有の独自ドメイン配下（サブドメイン違い可）で配信する。`*.onrender.com` 等のサービスドメイン単独での申請は避ける。
- **Page Access Token は長期トークンを使用**：審査期間中は短期トークンが失効しないよう、Never-expiring Page Token に変換して保持する。
- **テストアカウントは 2FA・IP 制限なし**：レビュワーがテストアカウントでログインできるよう、2FA と IP 制限は審査期間中は無効化する。
- **DB は Supabase（東京リージョン、無料プラン）**：個人運用の月額コスト最小化のため AWS RDS ではなく Supabase Postgres を採用する。Pause 回避のため keep-alive Lambda を運用する。
- **モックアップ UI の流用範囲**：`mock/` ディレクトリの受信トレイ・スレッド画面デザインは MVP 審査版の UI 参考として流用するが、**mock に含まれる以下の UI 要素は MVP では一切実装せず、審査用デモにも登場させない**（含めると「申請対象外機能を主用途に見せている」と Meta に判定されるリスクがある）：
  - 顧客管理画面、VIP タグ、購入履歴、顧客メモ、優先度
  - 商品管理画面、商品マスタ
  - カテゴリフィルタ（`price` / `intent` / `detail` / `shipping` / `stock` / `other`）
  - **AI 自動分類カテゴリの表示**（AI 下書き生成は MVP に含むが、カテゴリ分類は Phase 2）
  - Slack 連携 UI、通知設定画面
  - Instagram DM 表示、チャネル切り替え
  - トーン設定、カスタムプロンプト設定
  - 「Meta Business で続行」等の Supabase Auth 以外のログイン動線
  - 翻訳併記機能、多言語 UI
- **UI の正本**：spec.md と mock/ が矛盾する場合は **spec が正**。mock は将来の Phase 2/3 のビジュアル指針として残すが、MVP 実装の Source of Truth ではない。

## Out of Scope

以下は design v2 には含まれるが、**本 MVP 申請用アプリのスコープ外**である：

- AI によるメッセージ自動分類（6 カテゴリ）— Phase 2 で追加
- 顧客管理機能（VIP タグ、購入履歴、メモ、優先度）
- 商品マスタ管理
- Slack Incoming Webhook 通知
- Slack Block Kit による承認フロー
- Instagram DM 連携
- 24 時間窓外のメッセージ送信（Message Tags）
- PayPal/Wise 請求リンク自動生成
- 追跡番号自動返信
- Shopify 在庫・価格連携
- セルフサインアップ画面 / 新規テナント作成ウィザード（Phase 2）
- Stripe による月額課金統合（Phase 2）
- マルチオペレーター運用 / ロールベースアクセス制御（RBAC）— 1 tenant あたり 1〜2 ユーザー想定で MVP / Phase 2 では実装しない（Phase 3 以降で再検討）
- 外部 OAuth プロバイダ連携（Meta OAuth Federation 等）— Phase 2 以降
- リアルタイム更新（WebSocket またはポーリング／審査に必須ではないため手動リロードで代替可）
- AI 返信の完全自動送信（Human-in-the-Loop は MVP の必須要件）
- トーン／カスタムプロンプト設定
- 分析ダッシュボード

これらは App Review 承認後、同一 App ID 配下で Phase 2/3 として段階的に追加する。
