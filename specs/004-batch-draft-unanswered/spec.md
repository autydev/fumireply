# Feature Specification: 未返信メッセージのバッチ下書き生成

**Feature Branch**: `004-batch-draft-unanswered`
**Created**: 2026-06-20
**Status**: Draft
**Input**: User description: "短いメッセージを連続で送ってきたときに、最後のメッセージに対する返信しかされていない。最後の自分からの返信以降のメッセージをすべて読み取って、その文脈を踏まえた 1 つの AI 下書きが欲しい。"

## 背景と問題

現状、AI 下書きは **inbound テキストメッセージ 1 件ごとに 1 件** 生成される (`ai_drafts.message_id` が UNIQUE)。webhook が inbound テキストを受信するたびに `ai_drafts(status='pending')` を 1 行 INSERT し、`{ messageId }` を SQS に enqueue する。ai-worker はそのメッセージ時点までの履歴で 1 件の下書きを作る。

この設計は、顧客が短文を連投する運用 (例: スクリーンショットの「One piece」「Op-13」「Op-09」「And the single EB02-010」「And op-12 too」…) で次の問題を生む:

1. **無駄な下書きの量産**: 連投 K 件で K 件の下書きジョブが走り、各々がその時点までの部分文脈しか持たない。
2. **どの下書きが「正」か不定**: UI は「最新 inbound メッセージに紐づく下書き」を表示するが、最新メッセージが非テキスト (画像/スタンプ) の場合は下書きが付かず、古い文脈の下書きが表示される。下書きジョブの完了順も保証されない。
3. **バッチとして回答されない**: 利用者が本当に欲しいのは「自分の最後の返信以降に届いた未返信メッセージ群すべて」をまとめて読み取った 1 つの返信案。現状はメッセージ単位の生成のため、複数の問い合わせを 1 通でまとめて捌けない。

## スコープ

本 spec は上記**課題2 のみ**を扱う。下記は本フェーズのスコープ外とし、別 Issue で追跡する:

- **外部送信メッセージの取り込み** (Messenger アプリ等から運営者が送った返信を `message_echoes` で取り込む) → [#65](https://github.com/autydev/fumireply/issues/65)。本機能の「最後の自分の返信」境界の精度は、外部送信が DB に記録されて初めて完全になる (Assumptions / Dependencies 参照)。
- **条件付き再生成 UI** (運営者がワンオフ指示を付けて下書きを作り直す) → [#66](https://github.com/autydev/fumireply/issues/66)。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 連投された未返信メッセージを 1 通でまとめて下書きする (Priority: P1)

ショップ運営者は、顧客が短文を連続で送ってきても、最後の自分の返信以降に届いた未返信メッセージ群**すべて**を踏まえた 1 つの AI 下書きを受け取れる。個々のメッセージごとにバラバラの下書きが出るのではなく、会話に対して常に「いまの最新状態に対する 1 通の返信案」が提示される。

**Why this priority**: 連投は Messenger の主要な利用パターンであり、現状の「最後の 1 通にしか答えない」挙動は実運用の最大の不満点。これを解消しないと AI 下書きの採用率が下がる。単独で価値が出る最小機能。

**Independent Test**: 1 つの会話に対し、運営者の最後の返信以降に「Op-09 はいくら?」「EB02-010 も」「op-12 も」の 3 通を数秒以内に連投する。デバウンス経過後、会話に対して **1 件の下書き** が生成され、その本文が 3 件すべての問い合わせ (Op-09 / EB02-010 / op-12) に言及していることを確認する。

**Acceptance Scenarios**:

1. **Given** 顧客が最後の運営者返信以降に複数の inbound テキストを連投した、**When** 最後のメッセージからデバウンス時間が経過する、**Then** その会話に対して**ちょうど 1 件**のアクティブな下書きが生成され、未返信メッセージ群すべてに言及した本文になっている
2. **Given** 連投の途中段階 (まだメッセージが続いている)、**When** デバウンス時間内に次のメッセージが届く、**Then** 途中の下書き生成はスキップされ、最後のメッセージを起点とした 1 回の生成に集約される (無駄な生成が走らない)
3. **Given** 単発の inbound メッセージ 1 件のみ、**When** デバウンス時間が経過する、**Then** 従来同様にその 1 件に対する下書きが生成される (単発でも回帰しない)
4. **Given** 未返信バッチの最後のメッセージが非テキスト (画像/スタンプ) で、その直前にテキストがある、**When** 下書きが生成される、**Then** 直近のテキスト群を対象に下書きが生成され、非テキストのみで下書きが欠落することはない
5. **Given** 会話にアクティブな下書きが既に存在する、**When** 新たな inbound が届いてバッチが更新される、**Then** 会話のアクティブな下書きは 1 件に保たれ、最新バッチの内容に更新される (古いアクティブ下書きが重複して残らない)

---

### User Story 2 - 送信・破棄後は新着まで下書きを再提示しない (Priority: P2)

ショップ運営者が下書きを送信または破棄したら、その下書きは「消化済み」となり、次に新しい顧客メッセージが届くまで再提示されない。送信後にページを再読み込みしても、消化済みの下書きが復活しない。

**Why this priority**: 会話スコープの下書きに移行すると「いつ下書きが消えるか」の定義が必要。Story 1 の下書きライフサイクルを閉じる補完要件。

**Independent Test**: 下書きを送信し、画面を再読み込み (loader 再実行) する。送信済みの下書きが返信欄に再populate されないことを確認する。続いて顧客から新着メッセージを受信し、新しい下書きが生成されることを確認する。

**Acceptance Scenarios**:

1. **Given** ready 状態のアクティブ下書きがある、**When** 運営者が送信する、**Then** その下書きは消化済み (非アクティブ) になり、loader 再実行後も再提示されない
2. **Given** ready 状態のアクティブ下書きがある、**When** 運営者が破棄する、**Then** その下書きは消化済みになり、再読み込み後も復活しない
3. **Given** 下書きを送信または破棄した後、**When** 新しい inbound メッセージが届く、**Then** その新着を含むバッチで新しいアクティブ下書きが生成される

---

### Edge Cases

- **同時刻メッセージ**: 連投で複数メッセージの `timestamp` が同一秒の場合、(timestamp, id) の安定順序で「最新」を判定する。最新でないジョブはスキップされる。
- **デバウンス中の運営者送信**: デバウンス待機中に運営者が手動で返信を送った場合、後続の下書きジョブは「未返信メッセージなし」(全 inbound が最後の outbound より古い) を検知し、下書きを生成せずアクティブ下書きを消化済みにする (空振り下書きを出さない)。
- **未返信バッチの上限**: 未返信メッセージが極端に多い場合、対象は直近 `UNANSWERED_CAP` 件 (初期値 30) に制限する。文脈用の履歴は従来の `RECENT_MESSAGES_CAP` (50) を維持。
- **要約カーソルとの併用**: 003 の要約カーソル (`last_summarized_at`) による文脈圧縮はそのまま機能する。本機能は「未返信バッチ (= 回答対象) の明示」を追加するだけで、文脈ウィンドウ (カーソル以降) のロジックは 003 を踏襲する。
- **下書き生成失敗**: バッチ生成が API 失敗した場合、アクティブ下書きは `failed` になり、UI は再生成導線 (別 Issue) または次の新着での再試行を待つ。failed はアクティブ扱いせず、次の新着で新しい pending を作れる。
- **既存データの移行**: 移行前に存在する「メッセージ単位の ready 下書き」は、会話ごとに最新 1 件のみをアクティブとして残し、それ以外は `superseded` に倒す (data-model.md 参照)。
- **重複 enqueue / SQS at-least-once**: 同一ジョブが二重配信されても、coalesce 判定 (最新メッセージ起点でなければスキップ) と「アクティブ下書き 1 件」制約で冪等に吸収する。

## Requirements *(mandatory)*

### Functional Requirements

**下書きスコープの会話単位化**

- **FR-001**: System MUST AI 下書きを**会話単位**で管理し、1 つの会話に対してアクティブな下書き (status ∈ {pending, ready}) は最大 1 件とする。
- **FR-002**: System MUST inbound テキストメッセージ受信時に、対象会話のアクティブ下書きを `pending` に upsert し、起点メッセージ (anchor) を記録する。
- **FR-003**: System MUST 既存のメッセージ単位下書き (`ai_drafts.message_id` UNIQUE 前提のレコード) を、会話単位モデルへ移行する (移行スクリプトでデータを保全しつつ重複アクティブを解消する)。

**デバウンスによるバッチ集約**

- **FR-010**: System MUST inbound テキスト受信時の下書きジョブを、`DRAFT_DEBOUNCE_SECONDS` (初期値 20 秒) 遅延させて enqueue する。
- **FR-011**: System MUST 下書きジョブ実行時に、対象会話の最新 inbound テキストメッセージを再取得し、ジョブの起点メッセージが最新でない場合は生成をスキップする (後続ジョブが最終バッチを処理する)。これにより連投が 1 回の生成に集約される。
- **FR-012**: System MUST デバウンス・スキップ・生成の各判定を構造化ログに出力し、集約が機能していること (生成回数 << メッセージ件数) を観測可能にする。

**未返信バッチの抽出と下書き生成**

- **FR-020**: System MUST 「最後の outbound メッセージの timestamp」を境界として、それより新しい inbound テキストメッセージを**未返信バッチ**として抽出する。outbound が 1 件もなければ会話の全 inbound テキストが対象。
- **FR-021**: System MUST 未返信バッチが空 (= 直近の inbound がすべて最後の outbound より古い) の場合、下書きを生成せず、アクティブ下書きを消化済みにする。
- **FR-022**: System MUST AI へのユーザープロンプトで、未返信バッチを「回答すべき未返信メッセージ群」として明示し、**1 通の返信で全件に対応する**よう指示する。文脈 (要約 + カーソル以降の履歴) は 003 の合成を踏襲する。
- **FR-023**: System MUST 未返信バッチの対象件数を `UNANSWERED_CAP` (初期値 30) に、文脈履歴を `RECENT_MESSAGES_CAP` (初期値 50) に制限する。
- **FR-024**: System MUST 生成結果 (ready / failed) を、メッセージ単位ではなく**会話のアクティブ下書き**に書き込む。

**下書きライフサイクル (送信・破棄)**

- **FR-030**: System MUST 運営者が下書きを送信したとき、その会話のアクティブ下書きを消化済み (非アクティブ) にする。
- **FR-031**: System MUST 運営者が下書きを破棄したとき、その会話のアクティブ下書きを消化済みにし、loader 再実行後も再提示しない。
- **FR-032**: System MUST 下書きの送信・破棄後に新しい inbound が届いた場合、その新着を含む新しいアクティブ下書きを生成できる。

**UI 取得経路**

- **FR-040**: System MUST スレッド画面の下書き取得を「最新 inbound メッセージに紐づく下書き」から「会話のアクティブ下書き」に変更する。
- **FR-041**: System MUST 下書きの pending → ready ポーリングを会話単位で行う (`get-draft-status` の引数を messageId から conversationId に変更)。

**スコープ外 (明示)**

- **FR-OOS-001**: 外部送信 (Messenger アプリ等) メッセージの取り込み (`message_echoes` 購読 + outbound INSERT) は本フェーズのスコープ外 ([#65](https://github.com/autydev/fumireply/issues/65))。
- **FR-OOS-002**: 条件付き再生成 (ワンオフ指示を付けた手動再生成) は本フェーズのスコープ外 ([#66](https://github.com/autydev/fumireply/issues/66))。
- **FR-OOS-003**: 下書きのバージョン履歴 (複数案の保持・切替) は作らない。アクティブ下書きは常に最新 1 件で上書き。

### Key Entities

- **AI Draft (`ai_drafts`)**: 既存。本フェーズで以下を変更:
  - `conversation_id` を追加 (会話スコープ化)。`message_id` の UNIQUE を廃止し nullable 化 (anchor として保持)。
  - `status` の取り得る値に「消化済み (例: `dismissed`)」「世代落ち (例: `superseded`)」を追加。アクティブ = {pending, ready}。
  - 「会話ごとにアクティブ下書き 1 件」を partial unique index で保証。
- **Message (`messages`)**: 既存。スキーマ変更なし。「最後の outbound timestamp」「最新 inbound テキスト」「未返信バッチ」の抽出元。
- **Conversation (`conversations`)**: 既存。スキーマ変更なし (要約カーソル等は 003 のものを再利用)。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 未返信メッセージ K (2≤K≤10) 件を 1 つの会話に連投したとき、生成される下書きは **1 件**で、その本文が未返信バッチの各メッセージの論点に言及している (運営者の主観評価で 80% 以上のケースで「全件に答えている」)。
- **SC-002**: 連投 K 件に対する Anthropic 生成呼び出し回数が、平均で K 件から **1〜2 回**に削減される (デバウンス集約により従来比 60% 以上のコスト削減)。
- **SC-003**: 単発メッセージ 1 件のみのケースで、下書き内容・レイテンシともに従来と同等で回帰がない。
- **SC-004**: 下書きを送信または破棄した後に loader を再実行しても、消化済み下書きが再提示される割合は 0%。
- **SC-005**: 1 会話に対し同時に複数のアクティブ下書きが存在する状態が発生しない (DB 制約で 100% 保証)。

## Assumptions & Dependencies

- **「最後の自分の返信」境界の精度は別 Issue に依存**: 本機能は「最後の outbound メッセージの timestamp」を未返信バッチの境界に使う。fumireply 経由の送信は `messages(direction='outbound')` として記録されるため通常運用ではこの境界は正しい。ただし運営者が Messenger アプリ等から外部で返信した場合、その返信は現状 DB に記録されない (`message_echoes` 未購読) ため、未返信バッチに「すでに外部で答えたメッセージ」が混入しうる。これは **外部送信取り込み Issue** の解決で完全になる。本フェーズはこの既知の限界を許容する。
- デバウンスは SQS の `DelaySeconds` (最大 900 秒) を利用し、新規インフラを追加しない。
- ai-worker / SQS / プロンプト合成 (003) の基盤はそのまま再利用する。
- 既存の要約パイプライン (003) とは独立。要約カーソルによる文脈圧縮はそのまま機能する。
- 下書きは常に最新で上書き (バージョン履歴なし)。これはユーザー判断で確定済 (Resolved Decisions D-002)。

## Resolved Decisions

- **D-001**: 下書きスコープは **会話単位に転換** (メッセージ単位のまま維持しない)。理由: メッセージ単位 + デバウンスだけでは「どの下書きが正か」「孤児 pending の残留」が解消されないため。ユーザー判断で確定。
- **D-002**: 再生成・送信後の下書きは **最新で上書き** (バージョン履歴を持たない)。ユーザー判断で確定。
- **D-003**: 外部送信取り込み (課題1) と条件付き再生成 (課題3) は本 spec のスコープ外とし、別 Issue で追跡する。ユーザー判断で確定。
