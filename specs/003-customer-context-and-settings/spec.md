# Feature Specification: 会話コンテキストの永続化と設定の階層化

**Feature Branch**: `003-customer-context-and-settings`
**Created**: 2026-05-20
**Status**: Draft
**Input**: User description: "会話コンテキストの永続化と設定の階層化 — AI ドラフト生成プロンプトを「直近 N 件メッセージ + AI 要約」のハイブリッドにし、ページ単位と顧客単位の 2 階層で AI 動作をカスタマイズできるようにする。スレッド画面に CustomerPanel (右カラム)、サイドバーの未実装 Settings リンクを実ページ化する。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ページ単位の店舗ポリシーを AI ドラフトに反映する (Priority: P1)

ショップ運営者は Settings 画面で「店舗ポリシー (カスタムプロンプト)」を 1 枠の自由記述で設定でき、その内容が以降のすべての AI ドラフト生成に反映される。配送ポリシー、決済方法、価格基準、トーンの基本方針などを 1 つの自由記述で書き、Claude が読み取って返信に反映する。

**Why this priority**: ハードコードされたシステムプロンプト 1 種類でしか動かない現状の最大の制約を解消する。これがなければページごとのビジネスロジックが反映できず、AI ドラフトの実用性が一段下がる。MVP として最小単位で価値が出る。

**Independent Test**: Settings 画面でカスタムプロンプトに「すべての返信に "Thank you for shopping with Malbek TCG" を必ず含める」と書き保存し、新着メッセージに対する AI ドラフトに該当文字列が含まれていることを確認する。

**Acceptance Scenarios**:

1. **Given** ショップ運営者がいずれかの接続ページの Settings 画面を開いている、**When** カスタムプロンプト textarea に内容を入力し保存する、**Then** 保存成功が UI 上で確認でき、データが永続化される
2. **Given** ページにカスタムプロンプトが保存されている、**When** そのページ宛に新規 inbound メッセージが来て AI ドラフトが生成される、**Then** ドラフト本文がカスタムプロンプトの指示を反映している
3. **Given** カスタムプロンプトが空 (null)、**When** AI ドラフトが生成される、**Then** 既存のベースシステムプロンプトのみで動作し、回帰がない
4. **Given** サイドバーの「Settings」リンクをクリックする、**When** 画面が遷移する、**Then** 接続中ページの一覧とそれぞれのカスタムプロンプト編集 UI が表示される (現状の `href="#"` を実ページに置き換え)

---

### User Story 2 - 顧客 (会話) ごとのトーンと指示を AI ドラフトに反映する (Priority: P1)

ショップ運営者はスレッド画面の右カラム CustomerPanel で、その会話相手 (顧客) 専用の AI 動作を設定できる。トーンを 3 択 (フランク / プロ / 簡潔) から選び、カスタム指示 textarea に呼び方や特例 (例「ファーストネーム呼び捨て OK」「値下げ上限 -10%」) を自由記述する。内部メモは AI に渡さない運用上のメモとして別枠で残せる。

**Why this priority**: ページポリシー (P1) と並ぶ柱。顧客ごとの微調整ができないと、同じ店舗でも常連と新規で口調を変えられず実運用に乗らない。CustomerPanel UI そのものを新設するため、Story 1 とは独立に実装・テスト可能。

**Independent Test**: 任意のスレッド画面を開き、トーンを「簡潔」に切替、カスタム指示に「絵文字なし」と入力して保存。次に同じ会話で新着メッセージに対する AI ドラフトを生成し、絵文字を含まず簡潔な文体で出力されることを確認する。

**Acceptance Scenarios**:

1. **Given** スレッド画面が開いている、**When** 画面右側に CustomerPanel が表示される、**Then** 顧客名・PSID、AI ドラフト設定 (トーン 3 択ボタン + カスタム指示 textarea)、メモが見える
2. **Given** CustomerPanel でトーンを変更する、**When** 操作してから数秒待つ、**Then** 自動保存され「保存済み」表示に変わる
3. **Given** 顧客にトーン「簡潔」とカスタム指示が保存されている、**When** その会話で AI ドラフトが生成される、**Then** ドラフトの文体・指示がカスタム指示を反映している
4. **Given** 顧客の AI ドラフト設定が空 (null)、**When** AI ドラフトが生成される、**Then** ページポリシーとベースシステムプロンプトのみで動作する
5. **Given** 内部メモ欄に何か書かれている、**When** AI ドラフトが生成される、**Then** メモ内容はプロンプトに含まれない (運用者専用)

---

### User Story 3 - 会話要約による長期コンテキストの活用 (Priority: P2)

ショップ運営者にとって長い会話履歴 (数十件以上) を持つリピート顧客とのやり取りで、AI が「以前は Wise で決済した」「前回 Charizard を購入した」等の文脈を踏まえた返信を作れるようになる。要約は AI が自動で生成・更新し、人手介入なしで運用される。CustomerPanel に「AI が認識している顧客像」として表示され、運営者は AI がどう理解しているかを目視確認できる。注意書きで「会話から AI が推定した情報であり不正確な場合がある」旨を明示する。

**Why this priority**: トークン肥大の解消とリピート顧客対応の質向上に貢献するが、Story 1/2 がなくても AI ドラフトは動く。要約パイプライン (新着 N 件ごと / 夜間バッチ) の新規実装が必要で、Story 1/2 より工数が大きい。

**Independent Test**: 1 つの会話で要約カーソル以降の累計本文文字数が閾値 (初期値 2,000 文字) に達するメッセージをやり取りし、要約パイプラインが発火して `conversations.summary` が生成・保存されることを確認。次の AI ドラフト生成でプロンプトに要約が含まれ、過去メッセージ全件投入時と比べて入力トークン数が一定範囲内に収まることを確認する。

**Acceptance Scenarios**:

1. **Given** 会話の要約カーソル以降の累計本文文字数が閾値 (初期値 2,000 文字) に達した、**When** 要約パイプラインが差分発火する、**Then** `conversations.summary` が生成・更新され、カーソルが新しい時点に進む
2. **Given** `summary` がすでに存在する会話、**When** カーソル以降にさらにメッセージが追加され閾値に再度達する、**Then** 既存要約 + カーソル以降の全メッセージから新しい要約が生成され上書きされ、カーソルが再度更新される
3. **Given** `summary` を持つ会話、**When** AI ドラフトが生成される、**Then** プロンプトに要約が含まれ、それ以前のメッセージは原文として投入されない
4. **Given** CustomerPanel が開かれている、**When** `summary` が存在する、**Then** 「AI が認識している顧客像」セクションに本文と注意書き (推定情報の旨) が表示される
5. **Given** `summary` が空 (null)、**When** CustomerPanel が表示される、**Then** プレースホルダ (例「会話が短いため要約はまだ生成されていません」) が表示される
6. **Given** 要約パイプラインが API 失敗などで生成できなかった、**When** AI ドラフトが生成される、**Then** 要約なしでも従来 (直近 N 件のみ) でドラフトが生成され、ユーザー体験は止まらない

### Edge Cases

- **空状態**: 接続ページが 0 件の状態で Settings 画面を開いたとき、空状態のメッセージと「ページを接続する」導線が表示される
- **複数ページ**: 1 アカウントで複数ページが接続されている場合、Settings 画面はページごとにカスタムプロンプト枠を表示し、相互に独立して保存できる
- **自動保存中の離脱**: CustomerPanel / Settings で編集中に画面を閉じた場合、最後の入力が保存待ち状態でも、明示的に保存トリガーが走るかユーザーに警告が出る
- **要約と直近メッセージの重複**: 要約カーソル (last_summarized_at / last_summarized_message_id) を境界とし、要約は「カーソル以前 (= 旧要約 + その後追加分すべて) を圧縮した内容」、AI ドラフトプロンプトの生メッセージ部分は「カーソルより新しいメッセージのみ」とすることで二重投入を避ける
- **長すぎるカスタムプロンプト**: ページカスタムプロンプトは 2000 文字、顧客カスタムプロンプトは 1000 文字を上限とする。入力中は残り文字数を表示し、上限を超える保存は UI 側でブロックする。サーバ側でも入力バリデーションを行い、超過は 400 系エラーで拒否する
- **削除済み会話**: Meta データ削除コールバックで会話が削除された場合、要約・トーン・カスタム指示・メモも同時に削除される ([001 の deletion_log フロー] を踏襲)
- **トーン未選択**: トーン 3 択のいずれも選ばない初期状態 (null) を許可し、その場合はトーン指示なしで動作する
- **不正確な要約**: MVP では運営者からの手動操作 (再生成・クリア) を提供しない。次の閾値到達 (N 件追加) で自動的に上書き更新される。手動再生成 UI は次フェーズ (FR-OOS-006)

## Requirements *(mandatory)*

### Functional Requirements

**ページ単位カスタムプロンプト (Story 1)**

- **FR-001**: System MUST 接続ページごとに 1 つの自由記述カスタムプロンプト (テキスト) を保存できる
- **FR-002**: System MUST サイドバーの「Settings」リンクから到達できる Settings ページを提供し、接続中ページの一覧とそれぞれのカスタムプロンプト編集 UI を表示する
- **FR-003**: System MUST Settings 画面でカスタムプロンプトを入力すると自動保存し、保存状態 (編集中 / 保存中 / 保存済) を視覚的に示す
- **FR-004**: System MUST AI ドラフト生成時に、対象会話の所属ページのカスタムプロンプトがある場合、ベースシステムプロンプトの直後に挿入する
- **FR-005**: System MUST カスタムプロンプトが未設定 (null) のページに対して、回帰なくベースシステムプロンプトのみで AI ドラフトを生成する

**顧客 (会話) 単位の AI ドラフト設定 (Story 2)**

- **FR-010**: System MUST 会話ごとに「トーンプリセット」(列挙: friendly / professional / concise / null) を保存できる
- **FR-011**: System MUST 会話ごとに「カスタム指示」(自由記述テキスト) を保存でき、顧客固有の呼び方や特例を運営者が書き込める
- **FR-012**: System MUST 会話ごとに「内部メモ」(自由記述テキスト) を保存でき、AI に渡されないことを保証する
- **FR-013**: System MUST スレッド画面に右カラム CustomerPanel を表示し、顧客カード (アバター + 顧客名 + PSID)、AI ドラフト設定 (トーン 3 択 + カスタム指示)、メモ、AI 要約 (FR-021 参照) を含む
- **FR-014**: System MUST CustomerPanel での編集を自動保存し、保存状態を視覚的に示す
- **FR-015**: System MUST AI ドラフト生成時に、ページカスタムプロンプトの後ろに、対象会話のトーン (あれば) と顧客カスタム指示 (あれば) と要約 (あれば) を挿入する
- **FR-016**: System MUST 内部メモを AI プロンプトに含めない

**会話要約パイプライン (Story 3)**

- **FR-020**: System MUST 会話ごとに「AI 要約」テキストを保存できる
- **FR-021**: System MUST 「最後の要約生成時点以降に追加されたメッセージ本文の合計文字数 (inbound + outbound の双方向合算)」が閾値 (初期値 2,000 文字、運用調整可) に達したとき、要約を生成・更新する処理を起動する。メッセージ件数ではなく文字数を計測する理由は、分割送信を多用する顧客で早すぎる発火、長文 1 通を送る顧客で発火しない、というブレを避けるため
- **FR-022**: System MUST 要約生成は AI ドラフト生成パスとは分離した経路で実行し、ドラフト生成のレイテンシに影響を与えない
- **FR-023**: System MUST 要約はローリング方式で更新する。すなわち新しい要約は「既存の要約 + その後追加された全メッセージ」を入力として生成され、生成完了後は新しい要約 1 件のみが保持される。生成完了時点で会話のカーソル (last_summarized_at もしくは last_summarized_message_id) が更新され、以降の文字数カウントはこの新しい時点から再開される。これにより夜間バッチや「未要約会話の取りこぼし」を考慮する必要がなくなる
- **FR-024**: System MUST 要約生成に失敗した場合でも、AI ドラフト生成は要約なしのフォールバックで動作する。失敗した会話はカーソルが更新されないため、後続のメッセージ受信時に同じ閾値判定で自動的に再試行される
- **FR-025**: System MUST AI ドラフトプロンプトを構築する際、要約が存在する場合は「要約カーソル (last_summarized_at / last_summarized_message_id) より新しいメッセージのみ」を生メッセージとして投入し、二重投入を避ける
- **FR-026**: System MUST CustomerPanel で要約本文と「AI が会話から推定した情報であり不正確な場合がある」旨の注意書きを表示する
- **FR-027**: System MUST 要約が未生成の会話に対しては、CustomerPanel に明示的なプレースホルダを表示する

**ドラフトプロンプト合成順序 (横断要件)**

- **FR-030**: AI ドラフト生成時のシステムプロンプトは以下の順序で合成される:
  1. ベースシステムプロンプト (現行ハードコード分)
  2. ページカスタムプロンプト (該当ページに設定があれば)
  3. 会話のトーンプリセット (設定があれば)
  4. 会話のカスタム指示 (設定があれば)
  5. 会話の要約 (存在すれば)
  
  その後にユーザーターンとして直近メッセージが投入される

**データライフサイクル (横断要件)**

- **FR-040**: System MUST Meta データ削除コールバックで会話が削除されるとき、その会話に紐づく要約・トーン・カスタム指示・内部メモも同時に物理削除する
- **FR-041**: System MUST ページ接続が解除されるとき、そのページに紐づくカスタムプロンプトを物理削除する

**多言語化 (横断要件)**

- **FR-050**: Settings 画面および CustomerPanel の UI 文言は、002 で導入された Paraglide JS の多言語化基盤に従い、ja / en で表示される

**スコープ外 (明示)**

- **FR-OOS-001**: 顧客 PII の構造化保存 (住所、電話番号、メールアドレス、氏名以外) は今フェーズのスコープ外
- **FR-OOS-002**: Wise / PayPal 等の請求 API 自動連携は今フェーズのスコープ外
- **FR-OOS-003**: CustomerPanel の統計 (注文数 / LTV / 平均注文 / 初回)、タグ (VIP / リピーター / 新規 / 要注意)、購入履歴、クイックアクション (Wise / 追跡) は今フェーズのスコープ外
- **FR-OOS-004**: Settings の翻訳併記グローバル設定、ワンクリック承認モード、ナイトモードは今フェーズのスコープ外
- **FR-OOS-005**: Instagram DM、Slack 通知、商品管理は引き続きスコープ外
- **FR-OOS-006**: AI 要約の手動操作 UI (CustomerPanel での「再生成」「クリア」ボタン) は次フェーズに送る。MVP では自動更新 (FR-021〜FR-025) のみとし、不正確な要約は次回の閾値到達時の上書きを待つ。

### Key Entities

- **Connected Page**: 既存。接続中 Facebook ページ。今フェーズで `custom_prompt` (自由記述、null 可) を追加。
- **Conversation**: 既存。顧客との 1 対 1 会話。今フェーズで以下を追加:
  - `summary` (AI 要約、null 可)
  - 要約カーソル (`last_summarized_at` または `last_summarized_message_id` 相当の単一フィールド、null 可) — 要約が対象としたメッセージ範囲の上限を示す
  - `tone_preset` (列挙、null 可)
  - `custom_prompt` (顧客固有の自由記述、null 可)
  - `note` (内部メモ、null 可)
- **Message**: 既存。要約パイプラインの入力データとして利用。スキーマ変更なし。
- **AI Draft**: 既存。プロンプト合成順序 (FR-030) の変更影響を受ける。スキーマ変更なし。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ショップ運営者が Settings 画面を開いてから店舗ポリシーを保存完了するまでに 2 分以内で完了できる (初回利用時)
- **SC-002**: スレッド画面でトーンを切り替えてから、次に生成された AI ドラフトに反映されるまで、運営者から見て「即座に反映された」と感じる体感 (1 メッセージ往復以内)
- **SC-003**: 要約カーソル以降の本文累計が 10,000 文字を超えるような長尺会話における AI ドラフト 1 件あたりの入力トークン数が、要約導入前 (全メッセージ生投入) と比較して 60% 以上削減される
- **SC-004**: 要約パイプラインが過去 24 時間以内に対象としたすべての会話のうち、95% 以上で要約生成に成功する (失敗は致命的でなく、フォールバックが効くこと)
- **SC-005**: ページ / 顧客のカスタムプロンプト指示が AI ドラフトに反映されている割合が、運営者の主観評価で 80% 以上 (運営者抽出サンプル評価)
- **SC-006**: サイドバーの Settings リンクをクリックしたユーザーの 95% 以上がエラーなく Settings ページに到達できる (現状は `href="#"` で何も起きない)
- **SC-007**: 既存の AI ドラフト生成機能 (001 由来) について、本機能導入後も回帰がなく、カスタムプロンプト / 顧客設定 / 要約が全て null の状態でも従来通りドラフトが生成される

## Assumptions

- 既存の AI ドラフト生成基盤 (`specs/001-mvp-app-review/contracts/ai-draft.md`、Claude Haiku 4.5、Anthropic API、AWS Lambda Worker) はそのまま再利用する
- 要約生成にも同じ Claude Haiku 4.5 モデルを使用し、別経路 (専用 Lambda / 既存 Worker の別エンドポイント / 既存 Worker の別キューのいずれか) で実行する
- 顧客名 (`conversations.customer_name`) は引き続き平文保存。本機能で追加する全テキストフィールド (`summary`, `tone_preset`, `custom_prompt`, `note`, ページ `custom_prompt`) も平文保存する (PII 列暗号化は別スコープ)
- 002 ブランチで導入された Connect Page 設計 (Page ID 手入力 + サーバ側トークン保持) を前提とする
- Paraglide JS による多言語化 (002 由来) は本機能の UI にも適用する
- 要約閾値は **文字数ベース** で、初期値は前回要約生成時点以降の inbound + outbound メッセージ本文合計が **2,000 文字** に達した時点で発火する。実運用で生成頻度・コスト・要約品質を見ながら調整する。閾値が小さすぎると要約 API 呼び出しコストが増え、大きすぎると直近メッセージ累積によるトークン肥大が解消されない。文字数はトークン数と厳密に一致しないが、発火判定は近似で十分であり、DB 側で `LENGTH(body)` で安く計算できる利点を取った
- 内部メモは将来「AI に渡すかをトグルで選べるようにする」可能性があるが、今フェーズでは「AI に渡さない」固定
- スレッド画面右カラム CustomerPanel はデスクトップ幅前提。レスポンシブ対応 (狭幅で折りたたみ等) は今フェーズの最低限 (隠す / 表示するトグル) のみ
- spec が既存実装と食い違う場合はコードを正として spec を同期する方針を踏襲する

## Resolved Decisions

以下は spec ドラフト時に [NEEDS CLARIFICATION] として残っていたが、ユーザー判断で解決済み。各 FR / Edge Case に反映済み。

- **D-001 (旧 NEEDS CLARIFICATION 1)**: 要約パイプラインは **文字数ベースの差分発火のみ** を採用し、夜間バッチは導入しない。要約は「既存要約 + 要約カーソル以降の全メッセージ」のローリング方式で生成するため、夜間バッチでカバーすべき「取りこぼし」概念が存在しない。閾値はメッセージ件数ではなく文字数 (初期値 2,000 文字、inbound + outbound 合算) で判定し、分割送信を多用する顧客と長文 1 通を送る顧客の双方で安定して動作させる。FR-021 / FR-023 に反映。
- **D-002 (旧 NEEDS CLARIFICATION 2)**: MVP では要約への手動操作 UI (再生成・クリア) を提供しない。理想形 (両方ボタン) は次フェーズに送る。FR-OOS-006 / Edge Cases「不正確な要約」に反映。
- **D-003 (旧 NEEDS CLARIFICATION 3)**: カスタムプロンプトの文字数上限は **ページ 2000 文字、顧客 1000 文字**。Edge Cases「長すぎるカスタムプロンプト」に反映。
